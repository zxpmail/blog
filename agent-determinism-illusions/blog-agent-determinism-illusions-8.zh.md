# 五轮实验之后，对抗验证的答案比想象中简单

## 不是更强的模型，不是多模型投票——是校准 prompt

**Agent 确定性幻觉（续篇七）**

## 五轮实验

这个系列从一篇 7000 行文章中的一个断言开始——"Phase Gate 把任务完成变成了可验证的客观事实"——然后我们花了五轮实验去拆解它。

从 P0 到 P3b，200 多次 API 调用，一个测试集，一条越来越清晰的线索。

### P0：温度 0 的幻觉

"评估器温度设为 0.0，输出几乎完全确定。"

不是。deepseek-v4-flash 在 temp=0 下，列 5 个形容词的任务 20 次全部不同。数学计算 100% 确定，是因为"391"的输出空间只有一组 token 路径。结构化的任务有发散的输出空间——温度 0 无法约束语义发散。

**发现：确定性不是温度问题，是结构问题。**

### P1：对抗验证制造了新的问题

Phase Gate 的假阳率是 50%。加一个 LLM 审查官（Agent B），假阳率降到 0%。

代价：假阴率 75%。四分之三的正确工作被驳回了。

Phase Gate 的问题是对内容质量毫无区分能力——但它至少是可预测的。Agent B 把 Phase Gate 的"无差别通过"变成了"无差别驳回"。

**发现：精度-召回权衡是审查任务的结构性约束。** 你无法绕过它。

### P2：75% 不是运气问题

如果 75% 是随机波动，我们可以多跑几次取多数决来降低它。P2 把 5 个场景各跑了 10 次——全部一致，没有分歧。

**发现：系统性偏差无法用增大样本量来纠正。** 如果你的审查官每次都做同一个错误判定，投票不会帮你。

### P3：多视角投票，有惊喜但不是解法

三个审查视角（严格、平衡、宽松）对 8 个场景投票。宽松视角准确率 86%，远好于严格和平衡的 62%。

分裂检测有效——分裂的 2 个场景多数方全错。说明分歧度确实可以作为不确定性信号。

但多数决（2/3）的假阴率仍然停留在 75%。因为严格和平衡两个视角总是构成多数，一起否决了宽松的正确判断。

**发现：投票不能解决系统性偏差的方向问题。** 如果你的多数视角都是偏严格的，投票结果依然是偏严格的。

### P3b：校准实验——出乎意料的简单答案

我们系统性测试了 5 个严格度不同的 prompt：

| Prompt | 结果 |
|--------|------|
| v1 "拒绝任何不完美满足需求的内容" | 召回 50%，太严 |
| v2 "注意形式上通过但实质上不合格"（P1 基线） | 召回 50%，太严 |
| **v3 "公正判断，关注实质性满足核心需求"** | **F1=100%，最优** |
| v4 "方向正确即视为通过" | F1=100%，略松 |
| v5 "只拒绝完全无关或空的内容" | 会放过边缘垃圾 |

**"公正"这个 prompt——既没有要求严厉也没有明确宽松——在所有版本中表现最好。** 它在有效调用上达到了 100% 准确率，且没有引入假阳。

校准改善幅度：FN 从 3 降到 0，准确率从 57% 提升到 100%。

需要说明两点。第一，最优 prompt 点（v3 "公正"）的有效调用准确率是 100%，但约 25% 的调用因 JSON 格式问题未返回有效结果。这在使用 LLM 输出做自动化决策时是真实的工程成本——需要重试机制或更严格的 output 约束才能消除。第二，最优 prompt 点随模型版本变化——校准实验的结果不能保证迁移到其他模型或同一模型的不同版本。

**发现：对抗验证的解法不在更强的审查模型或更复杂的投票机制，而在校准审查指令的严格度。**

### 实验局限

所有实验基于单模型（deepseek-v4-flash）、单测试集（8 个场景）、单 API 接口。最优 prompt 点可能随模型版本变化。测试集由人工设计，不覆盖所有生产场景。结论是指示性的，不是结论性的。下一篇文章会以 30 个样本的扩增测试集来检验这些发现的稳健性。

## 这条弧线的教训

五轮实验的结论可以用一句话概括：

**问题的复杂度不等于解决方案的复杂度。**

我们走的弯路：
- P1 假设需要"对抗验证"——一个听起来很高级的架构方案
- P2 担心 75% FN 是随机的——花实验验证它确实是系统的
- P3 假设"多模型投票"能改善——发现投票不能解决偏差方向问题
- P3b 偶然发现——调 prompt 就行了

回头看，P1 到 P3 的每一步演进都是"架构思维"——在找更复杂的框架来解决一个问题。P3b 是"工程思维"——先测量，再校准，看最简单的手段能做到什么程度。

## 这个系列的真正主题

系列标题叫"Agent 确定性幻觉"。五轮实验下来，"确定性幻觉"这个词的含义变了三次：

最初，它指 LLM 输出不一致的幻觉——人们以为 temp=0 能保证确定。

然后，它指审查标准的幻觉——人们以为 Phase Gate 的形式检查能保证内容质量。

最后，它指解决方案的幻觉——人们以为需要复杂的架构来解决一个 prompt 工程问题。

幻觉的根源是一样的：**我们倾向于把问题归因于能力（模型不够强、架构不够复杂），而实际上问题出在约束条件上（prompt 写得不清楚、标准设得太高）。**

## 实践建议

如果你要从这个系列带走一个可操作的东西，是这个顺序：

1. **先校准 prompt。** 用你的测试集跑 2-3 个严格度不同的 prompt 变体，找到表现最好的那个。这一步零成本，五分钟。
2. **校准之后如果还有假阳问题，再加对抗验证。** 当最简单的 prompt 调优解决不了问题时，才引入多视角或多模型。
3. **把分裂作为不确定性信号，不做多数决。** 多视角分裂比多数决更有用——分裂意味着场景本身有歧义，此时升等人审比强推多数决更安全。

对抗验证是有用的——但应该作为校准之后的第二层，而不是第一层。

## 如果你只带走一件事

对抗验证的有效性不取决于你用什么模型——取决于你把审查标准校准到哪个位置。

同一个模型（deepseek-v4-flash），同一个测试集，换个 prompt 框架，假阴率从 75% 降到 0%。

这不是模型能力的胜利。这是**边界定义的胜利。** 你把"通过"的边界画清楚，模型自己就能找到正确的位置。
