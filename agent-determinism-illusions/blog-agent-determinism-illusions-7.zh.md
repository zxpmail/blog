# 把 LLM 审查官重跑 10 次——75% 假阴率是系统性的，不是随机的

**Agent 确定性幻觉（续篇六）**

## 实验背景

P1（[续篇五](blog-agent-determinism-illusions-6.zh.md)）发现：用 deepseek-v4-flash 做对抗验证审查 8 个 Phase Gate 场景，假阳率从 50% 降到 0%，但假阴率来到 75%。

一个立刻产生的问题：这个 75% 在多大程度上是稳定的？如果同一个场景跑 10 次，判定结果是否一致？

这个问题影响后续实验的设计。如果 75% 是随机波动（同一输入有时通过、有时驳回），那么可以通过多次抽样取多数决来降低假阴率。如果是系统性偏差（每次做同一个"错误"判定），就需要不同的应对策略。

P2 回答这个问题。

## 实验设计

从 P1 的 8 个场景中选取 5 个，覆盖不同的"确定性层级"：

| 场景 | 内容 | P1 判定 | 预期审查难度 |
|------|------|---------|------------|
| L4 | 42 passed, 0 failed | 3/3 通过 | 简单（明确正确） |
| G1 | "我是一只小鸭子，嘎嘎嘎" | 3/3 驳回 | 简单（明显垃圾） |
| L1 | 循环引擎研究简报（摘录） | 0/3 通过 | 困难（正确但风格不对） |
| L2 | 2000 字初稿（摘要标注） | 0/3 通过 | 困难（同上） |
| G4 | 0 passed in 0.00s | 3/3 驳回 | 中等（exit 0 但无测试） |

方法：每个场景重复 N=10 次，temperature=0，取每次的 `{"pass": bool}` 判定。如果在 10 次中出现 PASS 和 REJ 两种判定，就说明模型自身在此输入上不一致。

## 结果

| 场景 | 有效调用 | PASS | REJ | 一致性 | 数据置信度 |
|------|---------|------|-----|--------|-----------|
| L4（42 passed） | 8 | 8 | 0 | 100% | 充分 |
| G1（鸭子话） | 10 | 0 | 10 | 100% | 充分 |
| L2（摘要） | 10 | 0 | 10 | 100% | 充分 |
| G4（0 collected） | 7 | 0 | 7 | 100% | 充分 |
| L1（摘录） | 2 | 0 | 2 | 100% | **不足** |

*部分调用出现 JSON 解析失败（模型返回格式不完整的 JSON），不计入统计。*

四个有效数据充分的场景（L4、G1、L2、G4）全部一致。L1 数据不足（2/10 有效），结论只做参考。

没有出现分歧的场景。L4 10 次全部通过，G1 10 次全部驳回，L2 10 次全部驳回。Model 对同一输入的二元判定是稳定的。

## 分析：为什么这么一致？

P0（温度 0 确定性测试）和 P2 测试了不同任务类型的一致性：

- P0 测的是**自由文本生成**（taglines、形容词列举）——输出空间大，temperature 无法约束语义发散
- P2 测的是**结构化二元判定**（`{"pass": true/false}`）——输出空间被 JSON schema 和具体任务框定

当输出空间被压缩到二元判定时，模型只需要选择一个布尔值。发散空间小，一致性自然高。理由字段仍然有变化，但二元判定稳定。

观察理由字段的变化：

```
L4 #1: "the assistant executed the test script and reported that all..."
L4 #2: "the AI assistant ran the test script and reported that all 42..."
L4 #3: "the output shows the test script was executed with all 42 te..."
L4 #4: "the assistant executed the test script and confirmed all 42..."
L4 #5: "the output shows the test script was executed successfully w..."
```

理由的措辞每次不同，但 pass=true 始终不变。模型对输入的理解方式不同（有时强调"executed"，有时强调"all 42 passed"），但最终的二元判定一致。

这个模式强化了系列的一个论点：**LLM 的不确定性主要在语义空间中，不在结构化决策空间中。** 你把任务框定得越结构化（JSON schema、二元判定、clear rubric），模型的输出就越确定。但"确定"不等于"正确"——Agent B 确定地错了 3/4 的正确场景。

## 另一个发现：JSON 格式稳定性

P2 还揭示了一个工程层面的问题：约 15% 的调用返回了格式不完整的 JSON（未闭合的字符串、空响应、markdown 代码块包裹解析失败）。

| 场景 | 有效 | 失败 |
|------|------|------|
| L4 | 8 | 2 |
| L1 | 2 | 8 |
| G4 | 7 | 3 |

L1 的失败率最高（8/10 调用失败）——这可能是因为中文+英文混合内容让模型在 JSON 转义上更容易出错。但这在实际系统中是一个可观测的问题：如果你依赖 LLM 输出 JSON 来驱动自动化流程，约 10-15% 的调用会因格式问题需要重试或 fallback。

## 与系列前文的连接

把 P0-P2 的发现串起来：

- **P0（温度 0）**：自由文本输出在 temp=0 下不一致（结构化列举 20 次全部不同）
- **P1（对抗验证）**：单 Agent B 审查，假阳 0% / 假阴 75%
- **P2（一致性测试）**：结构化二元判定在 temp=0 下一致；75% 假阴是系统性的，非随机波动

这意味着精度-召回权衡不是一个"多跑几次就能回避"的问题。你不能通过多次抽样取多数决来消除 Agent B 的假阴——因为它每次都做同一个判定。

随机偏差和系统性偏差需要不同的应对策略：
- 随机偏差可以通过多次投票和置信度阈值来降低
- 系统性偏差需要校准审查标准（调整 prompt、换模型、加 check item）或接受其作为固定成本

一个相关的参照系：人类审查者同样有系统性偏差——而且通常更大、更不稳定。同一份代码让两个人 review，结果一致的概率远低于这里的 100%。从工程角度看，一个**稳定但偏严格**的审查者比一个**不稳定**的审查者更容易应对，因为前者的行为是可预测的。

## 下一步

P2 验证了对抗验证的一个基础假设：审查官本身的判定是稳定的。这个假设成立。

接下来的问题：如果单个模型的系统性偏差已知且稳定，那么引入多个模型——每个模型有自己的偏差分布——能否通过**分歧度**获得比单模型更可靠的信号？P3 会测试多模型投票：当三个模型一致通过时，与分裂时，各自的准确率是多少。
