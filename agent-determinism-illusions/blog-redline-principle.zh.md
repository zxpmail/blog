# 红线法则:生产级 Agent 的收敛条件

## 一个问题,十一个实验

2026 年上半年,我跑了 11 组实验来回答一个问题:**怎么让 Agent 循环在生产环境里可靠收敛?**

实验覆盖了词汇重叠、温度 0、Phase Gate、embedding 分离、多模型质检权衡、人工兜底 Harness 的六刀自拆、成本灵敏度、SPC 异常检测、冷启动基线漂移、路由分类准确率和循环收敛条件。

11 组实验的脚本全部公开,可一键复跑。

这篇文章不是实验罗列,是实验指向的唯一结论:**
生产级 Agent 不需要更聪明的循环设计,它需要一条红线——一个"到这里必须停"的客观条件。**

## 实验:有红线 vs 无红线

为了验证这个结论,我跑了三组对比实验:

**实验 A:有红线(编译+测试通过)**
任务:生成一段 Python 代码,用编译和测试结果判断是否完成。

结果:简单和中等任务 1 步收敛。复杂的 1 步没通过——但系统知道它没通过,因为红线(编译错误)给了明确信号。

**实验 B:无红线(LLM 自我评判)**
任务:写一段产品文案,让 LLM 自己打分,自己决定是否"够好了"。

结果:8 轮迭代,自评全部是"1/10"。模型永远在改,永远不满意,永远不说完成了。如果没有硬截断,这个循环不自然停止——而且它自己知道自己做得不够好,但停不下来。

**实验 C:无红线但有硬截断**
任务:同文案,但第 3 步强制停。

结果:第 1 版 81 字,第 2 版暴涨到 565 字,第 3 版包含了思维链泄漏的文本。截断不是因为"完成了",只是因为"到数了"。

结论:

| 指标 | 有红线(编译过) | 无红线(自判) | 无红线+截断 |
|------|-------------|-------------|------------|
| 收敛? | 自动停止 | 不自停 | 硬停,非完成 |
| 输出质量可验证? | 可 | 不可 | 不可 |
| 能进生产线? | 可 | 不可 | 不可(需人工标注) |

## 整条线指向的结论

11 组实验的全部发现,落在这三句话里:

**1. 代码能验证的是符号层,不是语义层。**
词汇重叠、embedding、Phase Gate、SPC——全部只能验证"动作发生了",不能验证"结果是对的"。

**2. 模型越强,收敛越快,但语义层的红线不会消失。**
DeepSeek 比 qwen3 在代码任务上收敛率高得多(100% vs 58%)。但对于开放式语义任务——写文案、写分析报告——再强的模型也没有"完成了"的客观信号。模型只是在发散的循环里跑得更快,不是跑得收敛。

**3. 循环的价值不是让发散变收敛,是在模型能力刚好够但第一下没做对的边界情况里提供一个修复通道。**
收敛的决定性因素不是循环设计,是任务是否有客观可验证的停止条件。有,则循环有益;没有,则循环再长也不会自然停止。

## 生产级 Agent 的红线法则

基于以上发现,生产级 Agent 的设计可以归纳为三条规则:

**有红线的任务 → 自动收敛,进入生产管线。**
代码编译通过、Schema 校验通过、测试全绿——这一类有明确可验证产出的任务,循环可以自动跑,到达红线就停。

**没有红线的任务 → 必须设硬截断。**
开放式语义任务——写文案、写报告、写分析——不存在客观"完成"信号。不能依赖 LLM 自判来停循环。必须设硬截断,且截断后的输出不能自动进入生产流程。

**截断点的处理 → 标注"未经验证",转入人工队列。**
硬截断不是因为系统判断任务已完成,只是因为预算到了上限。截断点之后的输出,必须标记为"未经自动验证",送入人工审核通道,不能自动执行任何外部操作。

## 更本质的一句话

如果你仔细看,会发现红线法则不是"怎么让 Agent 做更多"——它是在定义"什么时候不让 Agent 继续做"。

这是整条线唯一不依赖实验数据、但所有实验数据都指向它的结论。

**生产级 Agent 的前提条件,不是它能做的事情更多,而是它不能做的事情被提前标清楚、在红线处能可靠停下。**

| 任务类型 | 收敛信号 | 红线 |
|---------|---------|------|
| 代码/可验证产出 | 编译+测试通过 | 宽松步数上限(正常情况1-3步) |
| 结构化编辑 | Diff归零 | 上限+人工确认 |
| 开放性语义 | 不存在 | 上限到了直接交人,不尝试自动修复 |
