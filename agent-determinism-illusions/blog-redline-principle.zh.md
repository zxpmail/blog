# 红线法则:生产级 Agent 的收敛条件

## 一个问题

怎么让 Agent 循环在生产环境里可靠收敛?

围绕这个问题,我跑了 11 组实验,覆盖了词汇重叠、温度 0、Phase Gate、embedding 分离、多模型质检权衡、人工兜底 Harness 的六刀自拆、成本灵敏度、SPC 异常检测、冷启动基线漂移、路由分类准确率和循环收敛条件。

这篇文章不长,只展示 3 组直接比较红线索效的实验。其余 8 组在附录脚本中以备查证。

## 核心实验:同一代码任务,有红线 vs 无红线

之前版本的实验有一个混杂变量:有红线时跑的是代码任务,无红线时跑的是文案任务——任务类型不同,不能直接归因于红线。此处修正。

**统一任务:** 生成 Python 函数,验证方式为运行测试并匹配预期输出。

**条件 A(有红线):** 编译+测试通过 = 停止。信号客观:代码跑过了,输出对了。
**条件 B(无红线):** LLM 自判"完成了" = 停止。同一套代码,同时后台运行真实的编译+测试来记录实际正确性。

三个任务,各跑 3 次,上限 8 步:

| 任务 | 条件 | 收敛率 | 平均步数 | 边际贡献 |
|------|------|--------|---------|---------|
| simple | 有红线 | **3/3 (100%)** | 1.0 | |
| simple | 无红线(自判) | 1/3 (33%) | 8.0 | **+67%** |
| medium | 有红线 | **3/3 (100%)** | 3.3 | |
| medium | 无红线(自判) | 1/3 (33%) | 8.0 | **+67%** |
| complex | 有红线 | **3/3 (100%)** | 1.0 | |
| complex | 无红线(自判) | 0/3 (0%) | 8.0 | **+100%** |
| **总计** | **有红线** | **9/9 (100%)** | **1.8** | |
| | **无红线(自判)** | **2/9 (22%)** | **8.0** | **+78%** |

**关键发现:** 自判失败的模式不是假阳性(说 YES 但代码错),而是**假阴性**(代码对了但说 NO)。模型写出了正确的代码,但自判不信任自己,持续迭代——然后改坏或走到上限。红线(编译+测试通过)消除了这个"不信任自己"的问题:代码正确 + 信号触发 = 立即收敛。

**红线的边际贡献 = +78% 收敛率。** 同一任务,信号类型是唯一变量——这个差值就是红线作为客观停止信号的因果效应。

### 关于"编译通过究竟在验证什么"

红线中的"编译+测试输出匹配预期"不是在验证"语法正确",而是在验证**需求级正确性**:函数 is_even(4) 必须返回 True,is_even(3) 必须返回 False。这和 Phase Gate 检查"文件存在"是本质区别——后者验证"动作发生了",前者验证"结果符合预期"。

对于开放式语义任务(写一篇分析报告),不存在等价的客观验证条件。这不是"更好的红线设计"能解决的问题——它是任务类型本身的限制。

### 三种红线的区分

实验暴露了一个核心概念缺失:当前所有"红线"都被混为一谈,但它们的验证能力和工程成本完全不同。

**格式红线** — 验证成本最低,验证能力也最低。
检查文件存在、exit 0、语法解析通过、JSON Schema 格式合规。
它验证的是"输出形式正确",不是"输出内容正确"。Phase Gate 和 SPC 属于这一类。

**需求红线** — 验证成本较高,验证能力中等。
检查编译通过、测试输出匹配预期、业务断言通过。
它验证的是"输出结果符合预期"。V2 实验中使用的就是这一类。需要人工写测试和断言——成本由任务的可验证性决定,不是由系统设计决定。

**语义层红线** — 不存在。
没有代码、没有断言、没有 Schema 能验证"这篇分析报告的逻辑自洽性"或"这段文案的情绪准确度"。
**这是一个未解决的问题。** 当前技术栈下,没有任何自动手段能可靠判断开放性语义任务的"完成"。

| 红线类型 | 举例 | 验证内容 | 成本 | 能否作为收敛信号 |
|---------|------|---------|------|---------------|
| 格式红线 | exit 0/文件存在/语法通过 | 形式正确 | 极低 | ❌(不能——Phase Gate 假阳率 50%) |
| 需求红线 | 编译+测试通过/断言通过 | 结果正确 | 中 | ✅(能——V2 实验 100% 收敛) |
| 语义层红线 | — | 逻辑自洽/质量达标 | — | **不存在**(开放问题) |

V2 实验用的不是格式红线,而是需求红线。红线的 +78% 收敛贡献,是在"需求线"这一层级测量的。如果换成格式红线(仅语法检查),收敛率不会高——因为代码能编译不代表逻辑正确。

以下两条规则基于这个区分。同时注意:**只有需求红线能作为收敛信号使用。格式红线不够。语义层红线不存在。**

## 生产级 Agent 的红线法则

**规则一:有客观收敛信号的任务 → 自动收敛,入生产管线。**
代码编译通过、Schema 校验通过、测试输出匹配预期——这一类有明确可验证产出的任务,循环自动跑,到达红线就停。

**规则二:没有客观收敛信号的任务 → 必须设硬截断。**
开放式语义任务——写文案、写分析、写报告——不存在客观"完成"信号。不能依赖 LLM 自判来停循环。必须设硬截断,截断后的输出不能自动进入生产流程。

**规则三:截断后的输出 → 标注"未经验证",入人工队列。**
截断不是因为任务完成了,只是因为预算到了上限。这个地方需要承认:这不是完整的工程方案,而是运维兜底。一个可用于生产的人工介入协议至少需要包括:积压熔断(队列深度超过阈值时降级)、审核上下文预处理(减少思维链泄漏等噪声)、判定结果反馈调优(人工判定反哺截断阈值)。当前实验不覆盖这部分设计。

## 循环的边界

数据还指向另一个结论:**循环的价值不来自通用的"让 Agent 改到对为止",而来自一个狭窄的边界条件:模型能力刚好够但第一下没做好的情况。**

有红线时,medium 任务平均 3.3 步、complex 平均 1.0 步——不是因为 complex 更容易,而是因为 medium 的边界判断(fizzbuzz 的"3→Fizz,5→Buzz,15→FizzBuzz")恰好落在模型的近因失误区。这种失误可以通过修复通道解决。概念性失误(完全理解错了需求)则不行。

当前实验未充分覆盖这一区别——语法错误 vs 逻辑错误的修复成功率差异,是一个值得独立展开的方向。

## 更本质的一句话

红线法则不是"怎么让 Agent 做更多"——它是在定义"什么时候不让 Agent 继续做"。

生产级 Agent 的前提条件,不是它能做的事情更多,而是它不能做的事情被提前标清楚、在红线处能可靠停下。

| 任务类型 | 收敛信号 | 红线 |
|---------|---------|------|
| 代码/可验证产出 | 编译+测试通过(需求级) | 宽松步数上限(正常1-3步) |
| 结构化编辑 | Diff归零 | 上限+人工确认 |
| 开放性语义 | **不存在** | 上限+人工,不尝试自动修复 |
