# 红线法则:生产级 Agent 的收敛条件
**——在可验证任务中,客观信号优于模型自判**

*本文结论的范围限于有客观验证标准的任务(代码、结构化输出、可断言结果)。对于开放式语义任务(写文案、写分析),红线法则不适用。*

## 一个问题

怎么让 Agent 循环在生产环境里可靠收敛?

围绕这个问题,我跑了 3 组直接比较红线索效的核心实验,另有 8 组辅助实验(词汇重叠、温度 0、Phase Gate、embedding 分离、多模型质检权衡、SPC 异常检测、冷启动基线漂移、路由分类准确率)覆盖了其他相关维度。所有脚本均在附录中公开。

## 核心实验:同一代码任务,有红线 vs 无红线

**⚠️ N=3,指示性结果,非统计显著。** 以下数据展示方向性差异,不是精确效应量估算。

## 核心实验:同一代码任务,有红线 vs 无红线

之前版本的实验有一个混杂变量:有红线时跑的是代码任务,无红线时跑的是文案任务——任务类型不同,不能直接归因于红线。此处修正。

**统一任务:** 生成 Python 函数,验证方式为运行测试并匹配预期输出。

**条件 A(有红线):** 编译+测试通过 = 停止。信号客观:代码跑过了,输出对了。
**条件 B(无红线):** LLM 自判"完成了" = 停止。同一套代码,同时后台运行真实的编译+测试来记录实际正确性。

三个任务,各跑 3 次,上限 8 步。N=3,展示分布不估算效应量:

| 任务 | 条件 | 收敛 | 步数 | 均值 |
|------|------|------|------|------|
| simple | 有红线 | [1,1,1] | 全部 1 步 | 1.0 |
| simple | 无红线(自判) | [X,OK,X] | [8,8,8] | 8.0 |
| medium | 有红线 | [1,4,5] | — | 3.3 |
| medium | 无红线(自判) | [X,X,OK] | [8,8,8] | 8.0 |
| complex | 有红线 | [1,1,1] | 全部 1 步 | 1.0 |
| complex | 无红线(自判) | [X,X,X] | [8,8,8] | 8.0 |

**方向:** 有红线的 9 次中全部收敛,无红线的 9 次中仅 2 次实际收敛(且都跑满了 8 步硬上限才由自判触发停止)。差异方向稳定,但 N=3 无法排除随机波动。

**自判失败模式:** 9 次无红线尝试中,出现了 0 次假阳性(自判 YES 但代码错)和至少 4 次假阴性(代码对但自判 NO 或从不触发)。模型写对了但不信任自己,持续迭代直到跑满上限。

**需注意的偏见:** 自判 prompt("你觉得好了吗")天然倾向于让模型说"还不够好"。换一种 prompt 格式(如"输出 FINISH 如果代码通过")可能改变无红线条件下的收敛率。当前对比描述的是"特定自判 prompt vs 编译信号"的差异,不能直接泛化为"红线 vs 无红线"的通用结论。

### 关于"编译通过究竟在验证什么"

红线中的"编译+测试输出匹配预期"不是在验证"语法正确",而是在验证**需求级正确性**:函数 is_even(4) 必须返回 True,is_even(3) 必须返回 False。这和 Phase Gate 检查"文件存在"是本质区别——后者验证"动作发生了",前者验证"结果符合预期"。

对于开放式语义任务(写一篇分析报告),不存在等价的客观验证条件。这不是"更好的红线设计"能解决的问题——它是任务类型本身的限制。

### 三种红线的区分

实验暴露了一个核心概念缺失:当前所有"红线"都被混为一谈,但它们的验证能力和工程成本完全不同。

**需求红线的一个诚实标注:** 当前实验中的需求红线(编译+测试输出匹配预期)依赖人类预先编写的测试断言。这不是系统自动完成——这是人类预先定义了可验证的收敛条件,Agent 在这个边界内运行。

"需求红线有效"等价于说"如果人类预先写好了完备的验收测试,Agent 能在 1-3 步内满足验证条件"。这本质上是一种劳动前置——把验证成本从运行期转移到了设计期。如果任务不存在可预先编写的完备验证集(大部分开放式语义任务),需求红线不可用。因此,"需求红线"不是通用方案,而是一种任务分类:那些可以低成本预先定义验证标准的任务,适合走自动收敛通道;其余任务不在本文讨论范围内。

**格式红线** — 验证成本最低,验证能力也最低。
检查文件存在、exit 0、语法解析通过、JSON Schema 格式合规。
它验证的是"输出形式正确",不是"输出内容正确"。Phase Gate 和 SPC 属于这一类。

**需求红线** — 验证成本较高,验证能力中等。
检查编译通过、测试输出匹配预期、业务断言通过。
它验证的是"输出结果符合预期"。V2 实验中使用的就是这一类。需要人工写测试和断言——成本由任务的可验证性决定,不是由系统设计决定。

**语义层红线** — 本文实验范围内未发现可靠方案。
现有方法(LLM-as-judge、多轮辩论、自洽性检查)在有限测试中显示假阳性率高或不可复现,无法构成工程意义上的收敛信号。
**这是一个未解决的问题。** 并非原则上不可解——但当前技术栈下没有已知的工程手段能可靠判断开放性语义任务的"完成"。

| 红线类型 | 举例 | 验证内容 | 成本 | 能否作为收敛信号 |
|---------|------|---------|------|---------------|
| 格式红线 | exit 0/文件存在/语法通过 | 形式正确 | 极低 | ❌(不能——Phase Gate 假阳率 50%) |
| 需求红线 | 编译+测试通过/断言通过 | 结果正确 | 中 | ✅(能——V2 实验 100% 收敛) |
| 语义层红线 | — | 逻辑自洽/质量达标 | — | **未发现可靠方案**(开放问题) |

V2 实验用的不是格式红线,而是需求红线。红线的收敛优势是在"需求线"这一层级观察到的。如果换成格式红线(仅语法检查),收敛率不会高——因为代码能编译不代表逻辑正确。

以下两条规则基于这个区分。同时注意:**只有需求红线能作为收敛信号使用。格式红线不够。语义层红线在当前技术栈下没有已知的可靠方案。**

## 生产级 Agent 的红线法则

**规则一:有客观收敛信号的任务 → 自动收敛,入生产管线。**
代码编译通过、Schema 校验通过、测试输出匹配预期——这一类有明确可验证产出的任务,循环自动跑,到达红线就停。

**规则二:没有客观收敛信号的任务 → 必须设硬截断。**
开放式语义任务——写文案、写分析、写报告——不存在客观"完成"信号。不能依赖 LLM 自判来停循环。必须设硬截断,截断后的输出不能自动进入生产流程。

**规则三:截断后的输出 → 标注"未经验证",入人工队列。**

截断不是因为任务完成了,只是因为预算到了上限。"转人工"不是完整的工程方案——它是运维兜底。以下是一份可用于生产的人工介入协议设计草案:

#### 积压熔断

人工审核队列的吞吐量上限是硬约束。当 Agent 产出速度持续超过审核速度时,系统不可持续——仿真(`scripts/handoff-protocol-sim.py`,产出 5 单/分 vs 审核 3 单/分,队列容量 50,2 小时)显示队列溢出率 34%。**注意:这是参数估算示例,不是实际生产数据。** 不同产出/审核比率下溢出率会大幅变化——当比率 ≤ 1 时系统稳定,比率 ≥ 2 时不可持续。实际部署需按自身吞吐数据校准参数。熔断机制:

- **水位线:** 队列深度 > 容量 × 80% 时触发降级。新任务不再进入修复循环,直接输出原始结果并标记"草稿模式"。
- **极限:** 队列深度达到容量上限时丢弃最低优先级任务(记录到熔断日志),优先保障高价值任务进入审核。
- **恢复:** 队列深度 < 容量 × 30% 时解除熔断,恢复正常。

核心观察:队列设计不主导系统稳定性——**Agent 产出速度与人工审核速度的比率才是决定因素。** 如果产出持续大于审核,任何队列都会满。要么减速(限制 Agent 并发),要么加速(更好的审核工具),要么接受溢出(承受业务损失)。

#### 上下文预处理

截断时生成的原始输出可能包含多步执行轨迹、思维链泄漏(实验 C 中发现的)等冗余信息,直接展示给审核员会拖慢决策速度。

预处理规则:

- 从执行轨迹中提取三个字段:最终输出、最后一次错误信息、尝试次数。不发送完整轨迹。
- 审核员界面只展示:任务描述 → 最终输出(高亮) → 截断原因(步数超限/自判假阴性/格式异常)。
- 判定后采集:通过/驳回 + 原因标签(代码错误/逻辑错误/格式问题/幻觉/不清楚)。原因标签按频率排序后,出现超过 3 次的类型应添加自动过滤规则。

#### 反馈调优

人工判定的结果不应该只消耗一次就丢弃。反馈环让截断参数随审核结果自适应:

- **滑动窗口:** 最近 10 单人工判定的通过率。
- **通过率 > 80%:** 截断太紧(大量正确任务被送审)。上调步数上限或放宽触发条件。
- **通过率 < 40%:** 截断太松(大量错误输出没被拦住)。下调步数上限或收紧触发条件。
- **40%-80%:** 维持——截断策略在正确区域,人工补漏而非兜底。

仿真(2 小时,4 种配置)显示:基线条件下(产出 ≤ 审核速度),反馈调优在 30-60 分钟内将截断阈值收敛到稳定值。"转人工"从此有可测量的行为指标——等待时间、溢出率、通过率——每一维都可以设 SLO。

**需诚实注明的风险:** 反馈调优的结构和缺陷二(闭环校准)中被批的"人工判定→入库→定时微调"是同构的——都是基于历史数据调整系统参数。缺陷二的风险(分布偏移、打地鼠效应)同样适用于此:如果业务需求变了,基于之前 10 单人工判定学到的"通过/驳回"分布可能不再适用。区别在于:这里调的是步数上限(一个标量,变化空间有限),不是 LLM 的 Few-shot 示例(高维空间,变化不可控)。上限调优的失效域比 Few-shot 窄,但不是零。

## 循环的边界——一个待验证的假设

当前实验暴露了一个值得追问但无法回答的问题:**循环的修复能力是否有边界?**

有红线时,medium 任务平均 3.3 步而 complex 平均 1.0 步——这一差异可能意味着 fizzbuzz 的边界条件(3→Fizz,5→Buzz,15→FizzBuzz)恰好落在模型的"近因失误区"(第一下写错了语法或边界,但模型理解需求),所以修复循环有效。complex 的 group_by 涉及数据结构操作,模型一步就写对了。

**但这个解释是 post-hoc 的。** N=3 无法排除随机波动。更本质的问题是:如果失误是概念性的(Agent 完全理解错了需求),重复修复循环是否还能有效?当前实验不回答这个问题,因为所有任务对模型来说都是"能力范围内的"。超出能力范围的任务没有包含在实验设计中。

一个值得独立展开的方向:构造语法错误 vs 逻辑错误的两类任务,对比修复循环的成功率——前者预期高,后者预期低且随迭代次数增加不改善。

## 更本质的一句话

红线法则不是"怎么让 Agent 做更多"——它是在定义"什么时候不让 Agent 继续做"。

生产级 Agent 的前提条件,不是它能做的事情更多,而是它不能做的事情被提前标清楚、在红线处能可靠停下。

| 任务类型 | 收敛信号 | 红线 |
|---------|---------|------|
| 代码/可验证产出 | 编译+测试通过(需求级) | 宽松步数上限(正常1-3步) |
| 开放性语义 | **不存在** | 上限+人工,不尝试自动修复 |
