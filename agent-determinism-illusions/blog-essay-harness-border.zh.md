# Harness 的边界

**——兜底层为何不能被 prompt 化**

> LLM 越强，Harness 是变薄还是变厚？
> 这个问题没有标准答案，取决于你如何定义 Harness。

*本文是"判与造"系列的第四篇。前三篇([判的疲惫](blog-essay-judging-fatigue.zh.md)、[从 show code 到 show idea](blog-essay-show-idea.zh.md)、[镜子照不到思想](blog-essay-mirror-no-thought.zh.md))反复撞到同一个边界:模型和验证器共享文本信道时,验证退化为 compliance,而"把 idea 钉回 code"是唯一的逃逸。这篇把那个逃逸放到一个工程名字下——Harness,并追问它的命运。*

### 一、为什么会有这个讨论

最近读了一篇阿里云的技术分享，讲的是数据研发场景下 Multi-Agent 系统的 Harness 工程实践。文章提出了一个核心公式：

> **Agent = Model + Harness**

其中 Harness 包含六大支柱：Identity、Orchestration、Context、Gate、Recovery、Evolution。作者的判断是：模型决定 Agent 的上限，Harness 决定其下限。

文章结尾留了一个开放性问题：Harness 是 AI 时代的 DevOps（长期存在），还是模型能力不足时的临时补丁（终将消失）？

这个问题值得展开聊一聊。

### 二、一个更宽泛的定义

在进一步的讨论中，有人提出了一个比原文更宽泛的 Harness 定义：

> **凡是夹在“人类/业务需求”和“LLM 不确定性”之间的那一层，无论轻重、无论动静、无论表现形式，通通都是 Harness。**

按这个定义，Harness 的表现形式极其多样：

| 形态     | 举例                                 |
| ------ | ---------------------------------- |
| 重代码    | Java 写的权限过滤器、SQL 防注入校验             |
| 轻代码    | 状态恢复逻辑、超时熔断策略                      |
| 胶水代码   | 把多个 Agent 的输出拼接成结构化结果              |
| 脚本     | CI/CD 流水线里自动验收 Agent 输出的测试脚本       |
| Prompt | System Prompt、few-shot 示例、元 Prompt |

这个定义的优点是**包容性强**，几乎囊括了所有“让 LLM 输出变得可控”的手段。但缺点也很明显——**如果一切都是 Harness，这个词就失去了解析力**。

一个更克制的表述可能是：Harness 指的是**那些围绕 LLM 构建的、用于保障输出质量与安全性的工程化手段**。它不必须是某个特定的模块或架构，而是一个功能性的描述。

### 三、两种对立的终局判断

关于 Harness 的命运，目前至少有两种合理的判断：

**判断一：Harness 是过渡品，终将消失**

逻辑很直接：LLM 的上下文窗口在膨胀（从 128K 到 10M），推理能力在提升（从 GPT-4 到 GPT-5/6），工具调用越来越精准。今天需要在代码里硬编码的“状态管理”“异常捕获”“格式校验”，未来模型自己就能搞定。

按照这个逻辑，Harness 会不断把“腐肉”（那些机械的、确定的、重复的工程逻辑）甩给 LLM，自身越来越薄，直到薄到看不见。

**判断二：Harness 是持久层，不会消失**

逻辑同样成立：只要 Agent 还需要落地到真实的物理世界——操作数据库、发起支付、控制硬件——就一定需要一层“兜底”。模型可以“建议”安全，但只有工程层能“强制执行”安全。

而且，模型变强并不意味着约束变少。恰恰相反，能力越强的 Agent，能做越多的事，也就越需要明确的边界（什么能做、什么不能做、做到什么程度算好）。这些边界本身就是 Harness。

### 四、一个更现实的看法：“移动”而非“消失”

一种更折中的看法是：Harness 不会消失，但它会**持续移动**。

移动的方向是**向上**——从底层的“机械执行”移动到高层的“目标治理”。

可以粗略地画一条迁移路径：

| 阶段            | Harness 的焦点 | 具体内容                      |
| ------------- | ----------- | ------------------------- |
| 现在（2024-2026） | 执行层         | 状态管理、格式校验、异常捕获、API 编排     |
| 中期（2026-2028） | 质量层         | 输出验收、自我反思、多 Agent 仲裁、合规红线 |
| 长期（2028+）     | 目标层         | 优先级排序、资源博弈、人类偏好编码、伦理约束    |

底层的“腐肉”确实会被 LLM 吃掉——那些繁琐的 JSON 解析、显式的状态枚举、硬编码的异常捕获，未来模型自己就能处理好。

但 Harness 会在新的层面长出“新肉”——当 Agent 从“执行指令”进化到“自主决策”时，如何确保它的目标与人类对齐、如何分配有限的算力资源、如何在多个 Agent 之间建立协作秩序——这些是新的工程挑战，而不是消失了。

**更准确的说法可能是：Harness 的形式在变，但功能永不消失。只要 Agent 还需要对物理世界产生实际影响，就需要一层“翻译层”把概率性的模型输出翻译成确定性的业务结果。**

### 五、一个需要承认的边界

需要坦诚地说，以上讨论有很强的**场景依赖性**。

* 如果你的 Agent 跑在**高容错的探索性场景**（比如内容生成、创意写作），Harness 确实可以极薄，甚至几乎没有。

* 如果你的 Agent 跑在**低容错的生产性场景**（比如数据研发、支付风控、SQL 执行），Harness 就必须足够厚重，以确保“准、稳、快”。

所以“Harness 会不会消失”这个问题，脱离具体场景是没有答案的。它更像一个**设计决策**——你需要为你的场景选择合适厚度的 Harness，而不是问“要不要 Harness”。

### 六、一个危险的方向：Harness 看起来在变“轻”

最后是一个需要警惕的观察。

模型越强，好的 Harness 设计确实**看起来**越轻。以前要写 200 行 Python 做状态管理，现在 System Prompt 里 3 句话描述状态流转，模型自己就能执行;以前硬编码一个 SQL 校验器，现在告诉模型"输出前自己检查一遍"。

这个观察本身没错——Harness 的**形式**确实在从代码层移向规则层。但这里藏着一个把"看起来轻"误读成"真的轻"的陷阱,而它正是我前面那几篇反复撞墙的同一个边界。

形式变轻,意味着 Harness 从"机器强制执行"变成了"请模型自觉"。代码层的 Harness 是环境兜底——SQL 注入校验器会拦住,不管模型想不想;prompt 层的"请自己检查"是文本层的请求——而文本层恰好是模型和验证器共享的信道。在这个信道上,"我检查过了"和"我声称我检查过了"在文本上无法区分。一旦 Harness 的实质从"环境强制"滑向"模型自觉",它就从兜底层退回到了自报告层——戴着绿色对钩的自报告。

所以"Harness 看起来变轻"不是它真的轻了,是它换了一种存在方式:**把看得见的代码强制,换成了看不见的依赖模型自觉**。前者厚重但可靠,后者轻盈但脆弱。一个 Harness 设计师真正该问的不是"能不能用 3 句 prompt 代替 200 行代码",而是"被替换掉的那 200 行,是不是其中有几行本来干的是环境兜底——那几行不该被 prompt 化"。

**这或许才是 Harness 真正的演化方向:从"显性的代码层"变成"隐性的规则层",从未缺席。但隐性不等于消失——把强制层误当成可省略的累赘,才是 Harness 设计里最危险的简化。**
