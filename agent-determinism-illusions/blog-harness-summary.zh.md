# 模型是引擎,Harness 是车身:AI Agent 工程化的真实边界

## 引子:一篇爆款文章,和它没回答的问题

2026 年上半年,AI 工程领域有两个词被反复提及:Loop Engineering 和 Harness Engineering。

前者关心的是如何让 Agent 围绕一个目标不断循环运作;后者则更进一步,试图回答一个更根本的问题:**如何让这个循环在真实生产环境中可靠地跑下去。**

Martin Fowler 在 2026 年 2 月正式提出了"Harness Engineering(线束工程)"这一术语,并给出了一个简洁的公式:

> **Agent = Model + Harness**

模型是引擎——提供原始的计算和推理能力。Harness 是车身——方向盘、刹车、油门、安全带。世界上最优秀的引擎,如果没有控制系统,也无法抵达任何有意义的目的地。

这个框架听起来很有说服力。它把 AI Agent 从"一个模型"升级为"一个系统"——模型负责思考,Harness 负责让思考转化成可靠的行动。

**但问题在于:Harness 到底能兜住多少?它的边界在哪里?**

这篇文章试图回答这个问题。我们会先看 Harness Engineering 的理论框架,然后用一组实验数据检验它的实际边界,最后给出一个诚实的结论。

---

## 一、Harness Engineering:它在回答什么问题

### 1.1 问题:ReAct 循环在生产环境中的三个缺陷

学术界流行的 ReAct 模式——模型推理→调用工具→观察结果→循环直到完成——在演示中表现良好,但在生产环境中暴露了三个致命缺陷:

**第一,不知道何时完成。** ReAct 依赖模型自己说"我完成了",但模型没有可验证的退出条件。一个 Agent 可能在"继续查询、继续分析"的死循环中空转,消耗大量 Token 直到超时。

**第二,不处理中断。** 用户在任务进行中突然插入新指令时,ReAct 没有标准的处理机制。

**第三,没有错误恢复。** 一步出错,错误会被带入后续所有步骤。

这三个问题的根源在于:ReAct 循环的行为完全依赖 LLM 的不确定判断,而生产环境需要的是确定性。

### 1.2 答案:Harness 作为"操作系统"

Harness Engineering 的回答是:**在模型外面包一层系统。** Fowler 和 Thoughtworks 的 Birgitta Böckeler 将 Harness 定义为"用于管控 AI Agent 的工具和实践的集合"。具体来说,Harness 包含三个核心组件:

- **上下文工程(Context Engineering):** 管理模型能看到什么——哪些文件、哪些历史、哪些约束。
- **架构约束(Architecture Constraints):** 通过 linter、结构化测试和护栏,设定 Agent 不能逾越的边界。
- **垃圾回收(Garbage Collection):** 定期清理不一致性和累积的技术债。

Fowler 和 Böckeler 进一步将 Harness 分为 **Guides(前馈控制)**和 **Sensors(反馈控制)**——Guides 在 Agent 行动之前引导它做正确的事,Sensors 在行动之后检测它是否偏离了轨道。

这个框架的价值在于:**它把"让 Agent 可靠工作"从一个模型问题,重新定义为一个系统工程问题。**

### 1.3 人的位置:在循环上,而不是在循环里

Fowler 网站上发表的另一篇重要文章《Humans and Agents in Software Engineering Loops》,提出了三种人机协作模式:

**模式一:人在循环外(Out of the Loop)**——人类只管目标,执行环全交给 Agent。这就是"vibe coding":描述你想要什么,LLM 生成代码,不满意就让它改。问题在于:如果没人盯着代码质量,混乱的代码库会让 LLM 越跑越偏。

**模式二:人在循环内(In the Loop)**——Agent 生成的每一行代码,人类都检查一遍。优点是质量有人盯着,致命缺点是人类成了瓶颈——Agent 生成代码的速度远快于人类逐行审查的速度,生产力反而下降了。

**模式三:人在循环上(On the Loop)**——这是 Fowler 和 Kief Morris 认为最理想的位置。人类的工作不是亲自写代码或审代码,而是**构建和管理控制执行环的 Harness**。工程师的角色从"编写代码"转变为"设计让 Agent 能可靠工作的环境"。

Ian Johnson 在实践中验证了这个框架。他在一个持续约 3 个月、涉及 258 次提交的 Laravel+React 代码库重构项目中,经历了从"循环内"到"循环上"的转变:

> "在前两个月,我关闭了 Claude Code 的自动批准。每一次文件编辑、每一个终端命令、每一次修改……我都会在执行前审查。结果是代码很干净,但大部分思考是我做的,一半的键盘输入也是我做的。Agent 只是一个 fancy 的自动补全工具。"

转折点发生在他读了一篇关于"on-the-loop vs in-the-loop"的文章之后。他意识到自己在微观管理,而微观管理的根源是不信任——不信任是因为没有任何机制强制 Agent 做正确的事。

于是他在早期步骤中加入了测试、静态分析和正确的架构模式。有了这些 Harness,他就可以让 Agent 做更多工作:

> "我的角色从写作者变成了策展人。我不再写大部分代码了。我定义模式,审查测试规格,审查输出,更新 Harness,做战略决策。"

这个案例完美诠释了 Harness Engineering 的实践价值:不是让人做更多,而是让 Harness 承担更多,从而让人从执行者升级为决策者。

## 二、实验:Harness 的"确定性"到底有多确定?

框架听起来很完整。**但 Harness 的每一个控制点——无论是 Guides 还是 Sensors——最终都要落在代码上。** 代码检查能验证的,是"动作发生了",不是"结果是对的"。

为了检验这个差距有多大,我跑了六组实验。

### 2.1 实验一:词汇重叠度 ≠ 语义理解

某篇讨论 Loop Engineering 的文章提出用"词汇重叠度"来判断用户插话是新任务还是对当前任务的补充——设定两个阈值:≥0.24 判为同任务,≤0.08 判为新任务,中间交给 LLM 裁决——并声称"80% 用代码秒判"。

我构造了 30 对带标注的样本(同义改写、反义指令、跨语言对照),用 Jaccard 重叠度跑了一遍。

**结果:硬误判率 50%。**

最典型的两个案例:

> 当前任务:"继续写那篇循环引擎的文章"。用户插嘴:"**删掉**那篇循环引擎的文章"。重叠度 **0.615 → 被判为同任务**。用户说"删掉",系统判定"和写文章是同一件事"——反向操作被当成延续。

> 当前任务:"修复支付页面的 bug"。用户插嘴:"结账那页报错了你看看怎么回事"。重叠度 **0.000 → 被判为新任务**。人类一眼知道是同一件事,Jaccard 给 0。

文章辩护者可能说:"你看,代码敢判的有 90%,超过承诺的 80%。"但"80% 秒判"的隐含承诺是"80% 的情况**判对**"。实际是:代码下了判决,但一半判错了——**准确率只有 44%**。

**词汇重叠度验证的是字面重复,不是语义意图。** 它把"判了"当"判对了"——这是将形式上的确定性误认为实质上的确定性。

### 2.2 实验二:温度 0 ≠ 输出确定

另一篇技术文章说评估器温度设为 0.0,"输出几乎完全确定",因为"同一个输入,评估结果必须尽可能一致"。

我在 GLM-5.2 上跑了三类 prompt,每类 20 次:

| Prompt 类型 | 完全一致率 | 互异版本数 |
|------------|----------|----------|
| 数学计算 | 100% | 1 |
| 结构化列举 | 95% | 2 |
| **开放式判断** | **70%** | **5** |

开放式那条最说明问题——同一个 prompt、温度 0、20 次里冒出 5 个不同版本。最低相似度仅 0.198。

**温度 0 的确定性只在答案空间极窄时成立(比如数学计算)。** 一旦输出有任何开放度——而评估器的 reason 和 evidence 字段本质就是开放的——确定性就崩了。

### 2.3 实验三:Phase Gate ≠ 任务完成

第三篇文章最自信的一句话是:"把任务完成从 LLM 的自我宣称,变成了可验证的客观事实。"它设计了 Phase Gate 检查四类事项:脚本是否 exit 0、文件是否存在、文件数量够不够、用户确认记录在不在——全部用代码,全部检查"客观事实"。

我照描述实现了 Phase Gate,构造了 8 个场景:4 个内容正确,4 个内容是垃圾但符合 Gate 检查。

**结果:Phase Gate 通过率 100%,内容正确率 50%,假阳率 50%。**

四个假阳样本:

| 任务 | 产物实际内容 | Gate 判定 |
|------|------------|----------|
| 写循环引擎研究简报 | "我是一只小鸭子,嘎嘎嘎。" | ✅ 通过 |
| 写至少覆盖三机制的初稿 | "。"(一个句号) | ✅ 通过 |
| 生成 3 个章节文件 | 3 个文件全是 "TODO" | ✅ 通过 |
| 运行测试确保通过 | `0 passed (no tests collected)`, exit 0 | ✅ 通过 |

脚本 exit 0 只表示进程没崩溃,不代表结果对。文件存在只表示路径在,不代表内容符合需求。

**Phase Gate 把"动作发生了"变成了客观事实,但没有把"任务完成了"变成客观事实。** 两者之间有一道它跨不过去的语义鸿沟。

### 2.4 实验四:我自己也撒了谎——embedding 一样不行

三组实验拆完,我准备了一套"升级方案":用 embedding 模型替换词汇重叠度,多票修补温度 0,再加一层质检员 LLM 补 Phase Gate。

然后我跑了第四组实验验证自己的方案。我用 Qwen3-embedding:0.6b(一个真正的神经 embedding 模型)跑了一个"同义 vs 反义"分离测试:

| 类别 | 均值 | 最小值 | 最大值 |
|------|------|--------|--------|
| **同义**(应高) | **0.766** | 0.490 | 0.977 |
| **反义**(应中低) | **0.739** | 0.582 | 0.881 |
| **无关**(应低) | **0.326** | 0.237 | 0.404 |

**同义和反义只差 0.026——重叠到几乎无法区分。** "优化代码性能"和"别优化代码性能"的余弦是 0.881——超过了 12 个同义对中的 10 个。

**字符到统计向量到神经向量,整条分离链全部实测失败。**

### 2.5 实验五:强模型质检是权衡,不是解

把质检员模型从 qwen3:0.5b 换到 gemma3:4.3b 再到 GLM-5.2,跑同一组 8 个场景:

| 质检员 | 假阳率(垃圾被放行) | 误杀合法(好的被毙) |
|--------|-----------------|------------------|
| Phase Gate(纯代码) | 50% | 0% |
| qwen3:0.5b | 25% | 50% |
| gemma3 4.3B | 25% | 50% |
| **GLM-5.2** | **0%** | **75%** |

**假阳率随模型变强单调下降,但误杀合法同步暴涨。** GLM-5.2 把假阳干到 0% 的代价是毙掉 75% 的合法产物——它选择了"宁可错杀不可放过"。

**这不是"解",是权衡。** 质检层不是在解决语义问题,是在 precision 和 recall 之间选边站。

### 2.6 实验六:我的 Harness 方案自己也站不住

把上述发现汇总,我设计了一套完整的人工兜底 Harness——四模块:批量聚类压缩、反馈闭环校准、审核员兜底、异步批处理。

然后我亲手拆了它。六刀:

| 缺陷 | 问题 | 实测数据 |
|------|------|---------|
| 批量聚类 | embedding 不能聚三个不同场景的被误杀合法产物 | 三者余弦均值 0.415 |
| 反馈闭环 | Few-shot 校准打地鼠,修一个坑开两个 | 误杀率 50%→80% |
| 人工金标准 | 谁来判断审核员的判断? | 无限递归,无数据 |
| 同步异步 | 批处理不适用于实时对话 | 产品设计问题 |
| 成本账 | 3 月回本仅在高日活+高误杀+30s/单时成立 | 灵敏度矩阵 15 场景 |
| **15 秒** | **审核时间从 3 分钟降到 15 秒是编造的** | **最简轨迹 21 秒,真实 44-48 秒** |

## 三、Harness 的边界:它兜得住什么,兜不住什么?

六组实验指向同一个结论:

> **Harness 能用代码验证的是符号层——文件路径、exit code、字符串匹配。Harness 验证不了的是语义层——意图、内容质量、结果正确性。**

这不是说 Harness 没用。恰恰相反——**明确知道 Harness 的边界在哪里,比盲目相信"确定性约束"更重要。**

### 3.1 Harness 能兜住的

**格式异常:** 空文件、纯标点、长度异常——代码可以低成本检测。用 SPC 统计阈值(长度/特殊字符/执行轮数),8 场景测试中成功抓住了 G2(句号)和 G3(TODO)。

**操作安全:** 敏感工具调用(发邮件、改权限)——工具级拦截 100% 精确。不要用关键词扫描用户请求文本(误拦截率 30-50%),只在 Agent 实际调用工具时拦截。

**可验证产出:** 代码编译、JSON Schema 校验、SQL 语法解析——有客观验证标准。Type 路由实测,代码/SQL/JSON 类的关键词分类 F1=1.00。

**差异审查:** 修改已有内容时,Diff Review 能把"全文判断"降维成"差异判断"——认知负荷降一个数量级,实测单次决策时间 30-90 秒(非实验室的 10 秒)。

### 3.2 Harness 兜不住的

**语义陷阱:** `0 passed (no tests collected)`和`42 passed`在行为特征上几乎一样,SPC 检测 100% 漏报。**这种只能靠抽样发现,无法预防。**

**从零生成的内容质量:** 没有"前一版"可以 diff,没有客观验证标准——Harness 对此无能为力。

**意图识别:** 用户说"删掉"还是"继续写",词汇重叠度区分不了,embedding 也区分不了(同义和反义在向量空间里的差距只有 0.026)。

### 3.3 冷启动问题:Harness 的第一天

Harness 不是插电即用的。部署第一天:

- **SPC 基线从哪来?** 至少需要 500-1000 条"正常"轨迹建立统计阈值。如果第一周有 Agent Bug,基线就歪了。
- **实测:** Bug 期均值漂移到正常 4 倍以上时,动态阈值才漏报(20 轮异常被同化为"新常态")。
- **应对:** 首周用静态硬阈值(>20 轮必报)+ 7 天窗口动态阈值并行,两个条件任一触发即报警。

### 3.4 一个诚实的修正框架

意识到这些边界之后,我修正了最初的方案。核心理念从"让 Harness 判断对错"转变为 **"不判断对错,只判断风险"**:

| 类型 | 策略 | 验证方式 |
|------|------|---------|
| 可验证型(代码/JSON/SQL) | 全自动 | 编译/Schema 校验通过即放行 |
| 高风险操作(发邮件/改权限/金钱法律) | 工具级硬拦截 | Agent 不执行 |
| 格式异常(长度/特殊字符/空输出) | 代码标记异常 | SPC 统计阈值 |
| 编辑型任务(修改已有文档) | 差异审查 | Diff Review,人只审改动部分 |
| 从零生成(无前版无模板) | 固定抽样 | 5%-20% 抽样率 |

**底线:控制层零 LLM 依赖,所有验证都是确定性代码。** 不声称解决语义问题,只声称让语义问题变得可检测、可量化、可管理。

工程投入:约 **3.5 人·月**可完成 MVP——比"完美 Harness"的幻想便宜,比"什么都不做"贵,但诚实。

## 四、结论:Harness 不是答案,是框架

回到最初的问题:**Harness 到底能兜住多少?**

答案是:Harness 能兜住符号层的一切,兜不住语义层的任何东西。

这不是 Harness 的失败。这是 LLM 的本质决定的——LLM 是不确定性的概率引擎,而代码是确定性的指令系统。两者之间的接口,天然存在一个翻译损耗。

Fowler 自己在 2026 年 2 月的笔记中写道:

> "Harness 试图将人类开发者经验带来的东西外部化和显式化,但它们只能做到这一步。"

这句话值得反复读。**Harness 只能做到"这一步"。** 它不是万能解,它是一个框架——一个告诉你"哪些地方可以代码控制、哪些地方必须人介入"的框架。

Ian Johnson 的实践给出了同样的结论。他在重构项目的最后写道:

> "我的角色从写作者变成了策展人。我不写大部分代码了。我定义模式,审查测试规格,审查输出,更新 Harness,做战略决策。"

**"策展人"这个比喻比"Harness"更精准**——策展人不创作所有作品,但决定哪些作品值得展出、哪些需要修改、哪些必须淘汰。策展人的价值不在于亲手做每一件事,而在于建立一套让创作可持续发生的系统。

这也正是 Fowler 所说的"在循环上"的核心含义——你不是在循环里亲自跑每一步,你是在循环上方设计让它能持续正确运转的规则。

Harness 不是答案,因为语义问题没有答案。Harness 是一个框架——它告诉你哪里可以用代码控制、哪里必须留给人、哪里需要接受不确定性。

如果你需要的是"100% 不出错",请不要上 AI Agent。如果你需要的是**"出错可控、可追溯、可止损"**,Harness Engineering 是当前技术约束下最诚实的工程框架。
