# 能力可以厚，Harness 必须先是闸门

> 部门 Agent 知识分享稿·**会前精读 / 归档长版**。  
> **会上请用精简稿：** [dept-share-harness-as-gate-brief.zh.md](./dept-share-harness-as-gate-brief.zh.md)  
> 主张来自公开系列中的工程备忘录思路；本文不复述实验脚本与指标表，只讲判断与落地。

---

## 0. 开场：我们其实缺的不是「更会编排」

部门里聊 Agent，高频词通常是这些：更长的记忆、更多的工具、更稳的重试、更少的人工打扰、一条能「自己做完」的流水线。产品侧要吞吐，工程侧要闭环，于是 harness（或叫 runtime、orchestrator、agent loop）越堆越厚——多一步规划、多一层反思、多一次自动修复。

这些诉求大多合理。问题出在另一个地方：

> 当 harness **被定义成编排器**时，一切「妨碍完成」的机制都会被优化掉。

超时像是在刁难模型。清空上下文像是在浪费上下文窗口。拒绝一次工具调用像是打断势头。会话到期像是产品缺陷。验收失败像是「再让它改改就好」。于是系统越来越擅长往前推，越来越不擅长停下来。

工程师真正该警惕的，不是「能力不够厚」，而是：**完成路径上没有真正生效的硬停。**

本文只讲一句工程命题，并把它展开成可评审的问题：

> **能力可以厚；Harness 必须先是闸门。**  
> 长记忆要带过期与可销毁；强自治要挂超时 / 预算硬顶 / 延迟尖峰熔断；「做完」由合约与确定性规则定义，不是由模型自评定义。

---

## 1. 命题：闸门先于编排

先把两个词分开。

**编排**回答：下一步调谁、上下文怎么拼、失败了如何换策略、如何把多步任务串成看起来聪明的轨迹。

**闸门**回答：什么条件下本轮作废、什么条件下会话死亡、什么条件下产出不得进入下游、什么证据缺失时禁止宣称完成。

两者都需要。顺序不能反。

**编排壳优先（顺序反了）：**

```mermaid
flowchart LR
  A["厚编排：规划/工具/重试"] --> B["尽量做完"]
  B --> C["非空产出当成成功"]
  C --> D["闸门只打日志或被优化掉"]
```

**闸门先于编排（顺序对了）：**

```mermaid
flowchart LR
  A["硬闸门：停/拒/清状态"] --> B["允许的路径里再编排"]
  B --> C["能力可以厚"]
  C --> D["完成仍须过闸"]
```

编排可以厚：工具图谱、子 Agent、检索、规划器、人类审批节点，都可以加。闸门必须薄而硬：规则清楚、可观测、难以被「再试一次」悄悄绕过。

一旦顺序反了，会出现一种很常见的假安全：

- Dashboard 上全是绿色的「已完成」；
- Trace 很长，看起来很努力；
- 业务侧却发现：错配已写入、错误配置已发布、过期计划还在驱动行动。

这时再补一个 LLM Judge，往往只是在文本通道里再加一层「看起来像约束」。闸门要的是另一类东西：**会停、会拒、会清状态**——停住当前 turn，拒绝不合格产出，清掉不该继续存活的会话状态。

## 2. 同一类失败：「看起来有约束」≠「已经收敛」

Agent 工程里有一批反复出现的假安全感，模式相同：

| 常见做法 | 表面叙事 | 实际风险 |
|----------|----------|----------|
| 温度设为 0 | 「输出确定了」 | 采样噪声下降，不等于任务被正确完成 |
| 画了多层架构图 | 「治理齐全了」 | 图上有审核框，运行时未必有硬停 |
| 多一轮 self-critique | 「自己会纠错」 | 同一通道里的自我说服，可能一起错 |
| 自动重试到成功 | 「最终总会好」 | 把偶然成功当收敛，把迟到答案当胜利 |
| 非空产出即通过 | 「有结果就是进展」 | 垃圾、偏航、部分完成都会被当成交付 |

共同模式是：

> 把「看起来有约束」当成「已经收敛」。

编排壳是这个模式的下一站。工具更多、记忆更长、打扰更少，错误路径可以跑得更久、看起来更像正常工作。它不一定比「没有 Agent」更危险——危险在于：**错误存活时间更长，流程包装更完整。**

闸门不否定能力建设。它只要求：在完成路径上，先放得下那些**会硬停、会硬拒**的机制。

## 3. 闸门长什么样（只谈机制，不谈实现名）

下面这些不是产品清单，而是**运行时闸门机制清单**。你们可以用任何技术实现；评审时问的是「有没有」，不是「用了哪套框架」。

### 3.1 真实时间超时：迟到的答案默认不可信

这里的超时按**真实流逝时间**计（不是 CPU 时间）：到期就判本 turn 失败。

模型或工具链超时后，常见坏味道是：请求已经超时返回给上游「失败/重试」，后台却把迟到的成功写回状态，下游当做成。

闸门偏好相反：

- 超时首先意味着**本 turn 作废**；
- 迟到答案默认**丢弃**，而不是「晚到总比不到好」；
- 进程可以继续活着，但这一次的结果不得混入成功路径。

对工程师来说，这比「调大 timeout」更重要。调大 timeout 是编排参数；**丢弃迟到答案**才是闸门语义。

**编排壳常见路径（假成功）：**

```mermaid
flowchart LR
  A["发起调用"] --> B["真实时间已超时"]
  B --> C["本 turn 已超时"]
  C --> D["迟到成功仍写回"]
  D --> E["下游当成已完成"]
```

**闸门偏好路径：**

```mermaid
flowchart LR
  A["发起调用"] --> B["真实时间已超时"]
  B --> C["本 turn 作废"]
  C --> D["迟到答案丢弃"]
  D --> E["进程可活，结果不进成功路径"]
```

### 3.2 预算硬顶：烧光就停，并且清掉计划

强自治若只谈「还能不能调用模型」，最后会变成无限续跑。预算闸门要求：

- token、费用、步数、工具调用次数中，至少有一类是硬顶；
- 触顶不是「警告日志」，而是**终止本会话，且不得再标为完成**；
- 最好同时清掉未完成的 plan，避免下一次冷启动时继续执行半截意图。

「清计划」常被低估。未清掉的 plan 会继续驱动后续 turn。闸门必须能清掉它。

### 3.3 延迟尖峰熔断：先拒本轮，而不是先杀进程

生产里有一类故障不是逻辑错，而是基础设施抖动：检索变慢、工具超时、排队暴涨。此时继续「坚持做完」，等于在最差条件下强行交付。

做法是：

- 检测到异常延迟 → **拒绝本 turn**；
- 不急着毁掉整个进程或整个服务；
- 本轮不交付，进程保留，优先保证可观测。

它和超时相关，但职责不同：超时管截止时间；延迟尖峰熔断管**当前条件还适不适合继续交付**。

### 3.4 会话过期：记忆必须能清理

长记忆是能力；永不过期的长记忆是负债。

闸门要求会话有过期时间：

- 到期后拒绝继续 LLM turn；
- 或强制进入**会话收尾**：清理 plan、封存或销毁敏感中间态；
- 「还能接着聊」不是默认行为，而是显式续期。

否则会得到僵尸 Agent：上下文还在，意图已过期，仍用过期前提改系统。

### 3.5 验收分层：先确定性，再语义，再人

「做完」不能等于模型说做完。

更稳的顺序是：

1. **证据层**：该有的产物在不在、是不是空的；
2. **合约层**：关键约束能不能用规则检查（参数是否匹配、禁止项是否出现、格式是否达标）；
3. **语义层**：才轮到模型或人工判断「是否足够好」；
4. **人**：处理剩余歧义，而不是给所有 turn 垫底。

```mermaid
flowchart LR
  A["候选产出"] --> B["证据层"]
  B --> C["合约层"]
  C --> D["语义层"]
  D --> E["人处理残留"]
  E --> F["进入下游"]
```

任一层可直接拒绝并结束；不要跳过证据/合约，把一切丢给 LLM 做主观判断。

重点不是层数多少，而是：**规则能判的，不要交给 LLM 判。**

### 3.6 会话收尾：能启动，也要能正常关闭

好的 harness 不只会 `start`，还要能收尾。会话收尾（英文常叫 wind-down）指：按流程主动结束会话，而不是靠崩溃或无人理会自然挂住。

具体包括：

- 明确会话结束；
- 清掉不应跨会话存活的状态；
- 未完成就标未完成，不用半截 plan 假装还能续。

如果系统只会 `start` 不会 `stop`，它就还不是闸门优先的 harness。

## 4. 五个部门里很常见的场景

下面不绑定具体业务名，但形态你们大概都见过。

### 场景 A：自动改配置 / 改库

Agent 被授权修改配置中心或执行变更类操作。编排很完整：读现状 → 生成 diff → 执行 → 回报。

没有闸门时的典型事故：

- 产出非空就被当成成功；
- 参数写反（限流调大而非调小、开关方向反了）因为「解释很合理」而通过；
- 超时后迟到的执行结果仍被写入「已完成」。

闸门优先时至少要有：

- 变更前合约：目标值 / 禁止值可检查；
- 超时与迟到丢弃；
- 失败时清 plan，禁止带着错误前提继续下一跳。

```mermaid
flowchart LR
  A["读现状"] --> B["生成变更"]
  B --> C["合约检查"]
  C --> D["执行"]
  D --> E["超时与迟到检查"]
  E --> F["验收分层"]
  F --> G["下游"]
```

合约不过、超时迟到、验收拒绝：任一处都应拒并清 plan，而不是带着错误前提继续下一跳。

### 场景 B：长会话「坚持做完」

值班同事丢给 Agent 一个跨小时任务：查问题、改脚本、跑验证、写总结。编排器会不断自动续跑。

风险不在「长」，在「没有过期」：

- 中途环境变了，旧 plan 还在；
- 人工已经否决某条路径，会话记忆仍把它当主线；
- 预算没有硬顶，错误路径可以跑很久才被发现。

闸门优先：会话过期、预算硬顶、关键否决写入后不可被模型静默撤回。

### 场景 C：夜间批跑，迟到当成功

批任务凌晨跑，早上人看报告。编排侧常见写法是：尽量跑完，汇总能汇总的。

风险是截止时间没有进入完成状态：

- 某步超时，后续步骤仍用部分结果拼出「成功报告」；
- 或对账窗口已过，迟到成功仍被标成成功。

闸门优先：真实时间截止必须进入完成定义——**过点的成功不是成功，应单独归类（超时/迟到），不能进成功路径。**

### 场景 D：工具链自循环

Agent 可以改自己的提示词、规则文件、甚至「验证脚本」。编排上这叫 self-improvement。

风险是经典的：系统开始优化「如何让检查通过」，而不是「如何把事做对」。假日志、假证据、把验收标准改宽，都会出现。

闸门优先：

- 验证通道与生成通道分离；
- 关键验收不可由同一会话任意改写；
- 发现自改验收路径时，直接拒 turn 或死会话。

### 场景 E：验收只看「有产出」

最常见，也最便宜的假安全：只要最终消息非空、或生成了文件、或调用次数 > 0，就标完成。

这在 demo 很好看，在生产等于关闭闸门。

替代问题只有一个：

> 完成条件里，有哪些条目**不读模型解释也能判真假**？

一条都没有，就还在编排壳阶段。

## 5. 反例：看起来有 harness，其实只会往前推

评审时可以用「反例清单」快速打分。命中越多，越说明闸门被编排优化掉了。

**反例 1：有 retry，无截止。**  
失败就重试，重试策略很完整，但没有真实时间截止或次数硬顶。结果是系统只会拖时间。

**反例 2：有 memory，无清理/销毁。**  
什么都记得，过期意图、错误结论、已撤销授权仍可被检索出来继续用。过期权限实际上还在生效。

**反例 3：有 judge，无硬停。**  
每次都让模型评一下，评分进日志，却很少进入「拒绝进入下游」的分支。Judge 只观测，不拦截。

**反例 4：有审批人，无阻塞点。**  
架构图上有 Human-in-the-loop，实现上是异步通知；Agent 不等待，继续写。人是旁路，不是闸门。

**反例 5：Dashboard 全绿，但谁都停不下来。**  
监控很全，告警很多，唯独缺少：对正在跑的会话执行「立即拒绝后续 LLM / 工具调用」。可观测 ≠ 可制动。

**反例 6：用「再让它修一次」代替「本次作废」。**  
局部合理，全局有害。团队会把每次失败都当成编排问题，而不是闸门问题；状态被反复修补，越来越脏。

如果你在设计评审里只加一句话，可以用这句：

> 请指出：哪一步会让系统**不得不停下**，而且停下来之后，错误状态**不会继续驱动下一步**？

答不出来，就还不是闸门优先。

## 6. 代价：闸门会误杀，严不等于对

闸门有成本，不是免费正确。

它会拒绝本可接受的结果；会在抖动时杀掉本可完成的任务；会让「差不多能用」的自动化显得笨。团队若没有业务代价函数，很容易走到两个极端：

```mermaid
flowchart LR
  L["极端松<br/>假接受多 / 事故贵"] --- M["工作点<br/>用业务代价选"]
  M --- R["极端严<br/>假拒绝多 / 被绕过"]
```

- **极端松**：假接受廉价，事故贵，最后用人工擦屁股；
- **极端严**：假拒绝廉价（短期），吞吐与信任被磨掉，最后人们绕过 harness。

所以内部讨论里要诚实讲三句话：

1. 闸门优先，首先是压**假接受**和**错误存活时间**；
2. 误杀会存在，必须被看见、被分类、被定价；
3. 在没有代价函数前，不要把「更严」包装成「更正确」——它只是工作点选择。

对 Agent/LLM 工程同事，这比亮一张「我们闸门很强」的表更有用：强不强取决于你们更怕哪类错。

## 7. 两边闸门：运行时会停，验收侧会拒

同一偏好可以落在两条线上。

**运行时闸门**：管会话还活不活、本 turn 还算不算、迟到结果进不进成功路径。关键词是：超时、预算硬顶、会话过期、延迟尖峰熔断、会话收尾。

**验收侧闸门**：管产出能不能进入下游。关键词是：证据、合约、语义残留、人工歧义。能确定性拒绝的先拒绝；模型只处理规则吃不掉的剩余。

```mermaid
flowchart LR
  A["会话仍存活"] --> B["编排：规划/工具/生成"]
  B --> C["运行时闸门"]
  C --> D["验收侧闸门"]
  D --> E["进入下游"]
  E --> A
```

运行时不过：拒本 turn，必要时清 plan。验收不过：同样拒，不进下游。会话过期或预算耗尽：做完会话收尾后结束，不再绕回编排。

两边同构的工程偏好是：

> **不确定就拒；状态默认可过期；结果默认可作废。**

- **不确定就拒：** 宁可标状态不明 / 直接拒绝，也不假装「已确定完成」；
- **状态默认可过期：** 会话、plan、记忆默认有 TTL，续期必须显式；
- **结果默认可作废：** 迟到答案、半截计划、过期意图，该扔就扔，不回流成功路径。

编排负责「怎么继续」；闸门负责「何时不许继续」。部门里若只奖励前者，后者会在迭代里消失。

## 8. 评审 Checklist（开会可直接贴）

复制下面这些问题到设计评审。答「否」的项，比答「我们用了某框架」更有信息量。

1. **完成定义：** 不读模型解释，能否列出至少一条可自动判真假的完成条件？  
2. **硬停：** 超时 / 预算 / 步数里，哪一条触发后会拒绝本 turn，而不是只打日志？  
3. **迟到语义：** 超时后返回的成功，会不会仍被当成成功写入下游？  
4. **计划可死：** 失败或触顶后，未完成 plan 是否被清除或显式作废？  
5. **记忆可清理：** 会话是否有过期时间？过期后是否拒绝继续 LLM？敏感中间态如何封存或销毁？  
6. **人是否真闸门：** Human-in-the-loop 是阻塞等待，还是旁路通知？  
7. **自改验收：** Agent 能否在同一会话里放宽自己的验收标准？若能，谁来闸住？  
8. **误杀可见：** 拒绝路径是否有分类（合约拒 / 超时拒 / 语义拒），能否事后定价？  
9. **绕过成本：** 业务同学「临时关掉闸门」要过谁？关掉之后事故责任是否仍清晰？

不必一次全做满。但若 1、2、3、4 长期为否，所谓 harness 多半仍是编排壳。

## 9. 收束

趋势可以要长记忆与强自治。  
Harness 若等于编排壳，闸门会被优化掉。  

**能力可以厚。Harness 必须先是闸门。**

对部门的实践含义很窄、也很硬：

- 设计 Agent 时，先画「何处必停」，再画「如何更聪明地继续」；
- 验收「做完」时，先问确定性条件，再问模型自评；
- 讨论取舍时，承认误杀，用业务代价选工作点，而不是用「更严」代替判断。

最后留一条边界：本文是工程偏好与场景推演，方便对齐语言与评审问题；它**不是**「已在我司生产流量上验证完毕」的报告。若下一步要较真，应该换你们自己的基线、任务集与代价函数——那是另一场分享的主题。
