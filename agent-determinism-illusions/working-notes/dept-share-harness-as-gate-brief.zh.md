## 1. 一句话命题

部门聊 Agent，常堆：更长记忆、更多工具、更稳重试、更少打扰。这些可以要。

问题在于：当 harness **被定义成编排器**时，超时、清上下文、拒绝工具调用、会话过期，都会被当成「妨碍完成」而优化掉。系统越来越会往前推，越来越不会停。

> **能力可以厚；Harness 必须先是闸门。**  
> 完成路径上先有硬停：超时、预算硬顶、会话过期、验收拒绝；「做完」由合约与规则定义，不是模型自评。

**编排**管：下一步调谁、怎么拼上下文、失败怎么换策略。  
**闸门**管：何时本轮作废、何时会话结束、产出何时不得进下游。

两者都要，**顺序不能反**。

```mermaid
flowchart LR
  A["硬闸门：停/拒/清状态"] --> B["允许的路径里再编排"]
  B --> C["能力可以厚"]
  C --> D["完成仍须过闸"]
```

反了会出现假安全：Dashboard 全绿、Trace 很长，业务侧却已写入错误配置或过期计划仍在跑。

## 2. 同一类失败

| 常见做法 | 实际风险 |
|----------|----------|
| 温度 0 | 噪声↓ ≠ 任务做对 |
| 架构图有审核框 | 运行时未必有硬停 |
| 自动重试到成功 | 偶然成功 / 迟到成功被当成收敛 |
| 非空产出即通过 | 垃圾和偏航也能「交付」 |

共同模式：把「看起来有约束」当成「已经收敛」。编排壳是下一站——错误存活更久，流程包装更完整。

## 3. 闸门机制（一张表）

评审问「有没有」，不问「用了哪套框架」。

| 机制 | 要什么 |
|------|--------|
| **真实时间超时** | 到期本 turn 作废；**迟到成功默认丢弃**，不进成功路径 |
| **预算硬顶** | token/费用/步数至少一类硬顶；触顶终止会话，并清 plan |
| **延迟尖峰熔断** | 基础设施抖动时先拒本轮，不强行交付 |
| **会话过期** | 记忆能清理；到期拒 LLM；续期必须显式 |
| **验收分层** | 证据 → 合约 → 语义 → 人；规则能判的别交给 LLM |
| **会话收尾** | 能 `start` 也要能 `stop`：清状态，未完成就标未完成 |

```mermaid
flowchart LR
  A["候选产出"] --> B["证据"]
  B --> C["合约"]
  C --> D["语义"]
  D --> E["人"]
  E --> F["下游"]
```

## 4. 两个典型场景

### A. 自动改配置 / 改库

无闸门：非空即成功；参数写反因「解释合理」通过；超时后迟到结果仍写入「已完成」。

要有：变更合约可检查；超时与迟到丢弃；失败清 plan。

```mermaid
flowchart LR
  A["读现状"] --> B["生成变更"]
  B --> C["合约检查"]
  C --> D["执行"]
  D --> E["超时/迟到检查"]
  E --> F["验收"]
  F --> G["下游"]
```

### B. 夜间批跑，迟到当成功

某步超时后仍用部分结果拼「成功报告」，或对账窗口已过仍标成功。

要有：真实时间截止进入完成定义——**过点的成功单独归类，不进成功路径**。

（更多场景见长版：长会话无过期、工具链自改验收、只看「有产出」。）

## 5. 反例速查

命中越多，越说明闸门被编排优化掉了：

1. 有 retry，无截止 → 只会拖时间  
2. 有 memory，无清理 → 过期权限仍生效  
3. 有 judge，无硬停 → 只打分，不拦截  
4. 有审批人，无阻塞 → 人是旁路通知  
5. Dashboard 全绿，却不能立刻拒后续 LLM/工具调用 → 可观测 ≠ 可制动  
6. 「再让它修一次」代替「本次作废」 → 状态越修越脏  

评审只问一句也行：

> 哪一步会让系统**不得不停**，且停后错误状态**不再驱动下一步**？

## 6. 代价：严 ≠ 对

闸门有成本：会误杀、会在抖动时拒掉本可完成的任务。

```mermaid
flowchart LR
  L["极端松<br/>假接受多"] --- M["用业务代价选工作点"]
  M --- R["极端严<br/>假拒绝多/被绕过"]
```

先压假接受和错误存活时间；误杀要可见、可分类、可定价。没有代价函数前，不要把「更严」包装成「更正确」。

## 7. 两边都要闸

```mermaid
flowchart LR
  A["会话仍存活"] --> B["编排"]
  B --> C["运行时闸门"]
  C --> D["验收侧闸门"]
  D --> E["下游"]
  E --> A
```

- **运行时：** 超时、预算、过期、熔断、会话收尾  
- **验收侧：** 证据 → 合约 → 语义 → 人  

偏好就三句：**不确定就拒；状态默认可过期；结果默认可作废。**

## 8. 评审 Checklist

1. 不读模型解释，有没有至少一条可自动判真假的完成条件？  
2. 超时/预算/步数触发后，是拒 turn 还是只打日志？  
3. 超时后返回的成功，还会不会写入下游？  
4. 失败或触顶后，plan 会不会被清除？  
5. 会话有没有过期？过期后还能否继续 LLM？  
6. Human-in-the-loop 是阻塞等待，还是旁路通知？  
7. Agent 能否在同一会话里放宽自己的验收标准？  
8. 拒绝路径有没有分类，事后能否定价？  

1～4 长期为否 → 多半还是编排壳。

## 收束

趋势可以要长记忆与强自治。Harness 若等于编排壳，闸门会被优化掉。

**能力可以厚。Harness 必须先是闸门。**

先画「何处必停」，再画「如何更聪明地继续」。
