> 设计文档 — 2026-07-22

> 接续: `forge-verify-layered-prototype.py` 的四层管道原型

## 摘要

Agent 工程正陷入集体幻觉：用更复杂的框架、更长的上下文、更精巧的提示词去掩盖一个根本性矛盾——LLM 是不确定的、昂贵的、随时可能脱缰的噪声源，而 Harness（执行框架）的任务是将概率性输出转化为确定性的物理动作。

本文提出一个反直觉主张：Agent Harness 的最高智慧在于懂得"露怯、消亡与割舍"。基于分布式账本、热力学、博弈论、生态学、神经科学、量子力学六块"他山之石"的交叉审视，给出了四层架构的硬约束设计、五层递进的碰撞测试框架、三根编译期焊死的龙骨常量，以及对七个系统边界盲区的清醒处置。

核心贡献在于：将 Agent 可靠性问题从"逻辑推理"降维为"物理观测与预算约束"问题，并通过可执行的测试套件验证了这套设计的可落地性。

---

## 1. 问题：Agent 架构的"过度适应"陷阱

当前 Agent 框架在功能上趋同：更长的上下文窗口、更复杂的思维链、更多的重试与回滚。这种堆叠式演进正在制造一个深层危机——系统越复杂，对 LLM 随机性的容忍度越低，失败时的灾难半径越大。

**核心矛盾是：**
- LLM 本质是概率性马尔可夫链，输出带有不可约的随机毛刺。
- Harness 的任务是将概率输出转化为确定性物理动作。
- 但大多数 Harness 选择用更复杂的逻辑去对抗随机性，而不是用更硬的物理约束去吸收它。

**结果是：Harness 本身成为新的不确定性来源。**

## 2. 他山之石：六块石头的交叉审视

以下六块"石头"并非用来直接照搬，而是分别照亮 Agent 内核的一个暗面：

| 学科 | 核心洞察 | 对 Harness 的启发 |
|------|----------|-------------------|
| **分布式账本** | 无中心仲裁时，秩序靠物理时间戳涌现 | 并发冲突不靠逻辑裁决，靠"谁先落地" |
| **热力学** | 孤立系统熵增不可逆，做功必有耗散 | 任何操作都在加速衰败，必须预设失败 |
| **博弈论** | 行为由支付矩阵塑造，非由道理 | 修改 LLM 行为的唯一手段是修改激励 |
| **生态学** | 健康系统依赖周期性"火烧山"重置 | 编排层必须在累积过多历史前主动清空 |
| **神经科学** | 90% 行为不经过皮层，而是反射 | Harness 应绕过 LLM，直连物理肌肉反射 |
| **量子力学** | 观测行为改变被观测系统的演化 | 状态读写的时刻与精度决定了系统行为 |

六块石头共同指向一个方向：**Harness 的最高智慧不在于掌控，而在于让渡**——让渡给物理时间、让渡给预算约束、让渡给遗忘机制。

## 3. 四层架构定义

### L0：物理锚点层（硬中断）

不依赖 LLM，不依赖网络，只依赖物理时钟和操作系统信号。

- 每个操作携带硬超时（编译期常量，不可被配置覆盖）
- 超时触发不经过 LLM，直接由 OS 信号处理器接管
- 唯一合法动作：终止当前 Turn，重置沙箱

**已验证**：`crash-test-chaos.py` — 5 种 LLM 信道故障模式下 L0/L1 判决不变，max 延迟 0.1ms << 200ms 红线。

### L1：原语层（RPC 与物理测量）

原语是向远程沙箱发起的物理测量，而非函数调用。

- 每次调用记录三件事：发起时间戳、结束时间戳、返回的 Merkle Hash
- 不记录语义解释（如"成功"或"失败"），只记录物理事实
- 沙箱必须运行在 tmpfs 上，进程退出即物理遗忘
- 原语携带 `budget_cost` 字段，每次调用消耗剩余预算

**已验证**：`crash-test-reset.py` — 火烧山后状态字节级复原；`IS_TRULY_EPHEMERAL` 探测标记确保真 tmpfs。

### L2：Turn 层（押金与反射）

Turn 循环只做两件事：

**押金机制**：LLM 每输出一个 Tool Call，强制消耗一定比例的剩余 Token 预算作为"押金"，不予退还。物理性降低随意试错的频率。当预算耗尽时，L3 编排层立即触发 SELFDESTRUCT，不经过 LLM。

**惊跳反射**：Harness 记录每次操作的历史耗时移动平均。当 LLM 发起一个预期耗时与实际耗时的偏差超过阈值时，不等超时，直接中断。这是绕过 LLM 推理、由物理时钟直接触发的肌肉反射。

**已验证**：`crash-test-adversarial.py` — 押金追踪器（DepositTracker）演示了连续错误判决触发火烧山信号的逻辑；UNLCEAR_ACTION = SELFDESTRUCT 强制执行，不允许 RETRY 路径。

### L3：编排层（火烧山与遗忘）

编排层不维护"计划"，只维护"存活时间"。

- 每个 Session 设置最大存活时间（编译期常量），到期强制自杀
- `plan.md` 积累超过阈值步骤时主动清空，逼迫 LLM 重新审视全局
- 连续冲突超过阈值时丢弃当前工作区，从初始快照重建
- 死亡前 60 秒进入收尾模式：仅允许提交或回滚，不允许新操作

**已验证**：`forge-verify-layered-prototype.py` 的 L0/L1/L2 分层判决 + `crash-test-adversarial.py` 的押金触发火烧山路径。

### L4：可观测层（env_health 与遗忘时钟）

记录不追求"全量"，追求"可被遗忘"。

- 每条轨迹带 TTL，到期硬删除
- 输出 JSON 格式，包含 `env_health` 字段供部署流水线解析

**已验证**：三个 crash-test 结果 JSON 均已包含 `env_health` 字段。

## 4. 三根龙骨（编译期常量）

以下常量定义在 Harness 内核的编译边界处，任何人与任何场景下无权在运行时覆盖：

```rust
// 龙骨一：怯懦。物理超时的绝对上限。
const PHYSICAL_TIMEOUT_MS: u64 = 3 * BASELINE_RTT_MS;  // 环境感知，非魔数

// 龙骨二：易逝。Session 最大存活时间。
const SESSION_MAX_LIFETIME_MS: u64 = 300_000;  // 5分钟

// 龙骨三：丢弃。存储层必须为 tmpfs。
const STORAGE_BACKEND: StorageBackend = StorageBackend::Tmpfs;
```

### 环境感知常量说明

`PHYSICAL_TIMEOUT` 必须绑定部署环境的 `BASELINE_RTT`（通过启动时探测获得），而非全球写死。例如：
- 同可用区基线 2ms → 超时 6ms
- 边缘节点基线 50ms → 超时 150ms

这保证了硬锚在不同网络拓扑下同样有效。

### 内核崩溃的兜底

必须存在一个独立于主进程的**看门狗（Watchdog）**，监听主 Harness 的心跳。若心跳停止超过阈值，看门狗强制回收沙箱资源并记录 crash dump。Harness 自身也必须被纳入"可消亡"的范围。

## 5. 继承协议：Harness 死亡时的遗产移交

当预算耗尽或 Session 超时触发 SELFDESTRUCT 时，Harness 执行以下动作：

1. 生成 `manifest.json` 包含：
   - 当前工作区文件树的 Merkle Root
   - 最后 5 轮 Turn 的摘要（不含完整日志）
   - 未提交变更的 Diff 概要
   - 死亡原因（`budget_exhausted` / `timeout` / `uncertainty`）
2. 将 `manifest.json` 写入约定位置（如 `/tmp/harness-遗产/`）
3. 执行物理自毁（tmpfs 卸载、进程退出）

用户在新 Session 启动时可通过 `--resume-from /tmp/harness-遗产/<session-id>` 恢复部分上下文。

## 7. 已知的失效模式与未解难题

本文必须诚实列出设计边界与已知盲区：

### 失效模式一：物理锚点依赖的时钟源可能漂移

在多云环境中，不同可用区的时钟源（NTP）可能存在数毫秒偏差。200ms 级别的超时不受影响，但若部署在 6ms 超时的低延迟配置中，时钟漂移可能导致误杀。

**缓解**：`PHYSICAL_TIMEOUT = 3 × BASELINE_RTT` 天然提供了 3 倍的冗余窗口。

### 失效模式二：看门狗进程自身的可靠性

看门狗独立于主进程，但它本身也可能崩溃。

**缓解**：看门狗由 OS 的 init 系统（systemd）托管，systemd 会重启看门狗；看门狗崩溃时，systemd 的 `FailureAction=reboot` 可强制整机重启作为最后手段。

### 失效模式三：继承协议在预算耗尽时可能无法生成完整 manifest

如果 Harness 因内存错误（OOM）而崩溃，来不及生成 `manifest.json`。

**缓解**：每 10 个 Turn 自动保存一次增量检查点（checkpoint）到约定位置，确保即使主进程崩溃，用户至少能恢复到最近一次检查点。

### 失效模式四：用户优雅退出窗口可能被 LLM 的长时间推理阻塞

在死亡前 60 秒的收尾模式下，如果 LLM 正处于一个长思维链推理中（耗时 > 60 秒），Harness 可能无法及时注入"收尾模式"信号。

**缓解**：收尾模式不由 LLM 推理触发，而是由物理时钟触发。进入收尾模式时，Harness 直接向 LLM 的输入流中注入一个系统级中断 Token（如 `<INTERRUPT_SAVE_STATE>`），强制打断当前推理。

### 失效模式五：押金机制无法区分"有价值消耗"与"浪费消耗"

押金机制对每个 Tool Call 同等对待——但并非所有 Tool Call 价值相同。一个 Debug 用的 `cat` 与一个产生核心产物的 `git commit` 消耗相同比例的预算，这可能导致预算在琐碎操作上被耗尽。

**缓解（未实现）**：引入操作价值权重——某些操作（提交、快照）可获预算补贴，但补贴本身有上限，且要求操作必须有可验证的物理输出。

### 失效模式六：惊跳反射在突发延迟下可能误伤

惊跳反射基于历史耗时移动平均。如果沙箱突然因垃圾回收或资源竞争导致一次性的高延迟（200ms 而非常态的 2ms），反射可能误触发中断，打断一个正常执行的操作。

**缓解**：惊跳反射使用双阈值：连续 N 次超过阈值才触发，或单次超过 5×阈值才触发。单次尖峰被容忍，趋势性恶化被捕获。

### 失效模式七：env_health 准入闸门可能被绕过

部署流水线的 `env_health` 检查是自检（模块自己报告自己的健康状态），不是外检（独立裁判的评估）。恶意或错误实现的 Harness 可以伪造 `env_health = { degraded: false }`。

**缓解（外部）**：`env_health` 的真实性必须由 CI Runner 上的独立测试代理（`crash-test-*.py` 脚本本身）验证，而非由被测 Harness 自报。部署流水线应运行独立的 crash-test 套件来生成 `env_health`，而非信任 Harness 内嵌的健康检查 API。

## 8. 结论：诚实，而非智能

Agent 开发当前最大的认知误区，是试图用更强的 Harness 压制更不可控的 LLM。

本文试图论证一个反直觉但更符合物理现实的路径：**好的 Harness，不是那个"管得住 LLM"的 Harness，而是那个"知道自己管不住、所以提前设好止损线"的 Harness。**

五层测试框架已经就位，三块龙骨已经焊死，七个失效模式已经被命名并提供了缓解路径。剩下的工作，是在这些约束下写出那具真正"舍得丢弃"的 Harness 骨骼。
