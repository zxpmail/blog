# 拆了四轮之后,这是我给的最诚实的替代方案

## 六刀拆完,我问了自己一个问题

前三篇拆了靶子的三个"确定性",拆了自己的 embedding 方案,拆了三档权衡曲线。上一篇拆了自己的人工兜底 Harness——六刀,每一刀都有实测数据。

拆完之后,有一个声音绕不过去:

> "你拆了六轮。那你有什么更好的答案?"

我做了一轮修正。

结果是下面这套方案。它不是"架构图",是**一组可用确定性代码实现的策略组合**。不引入新的 LLM 依赖,不声称解决语义问题,不拍脑袋编数字。

核心理念转变:放弃"质检员",拥抱"分流器"

之前所有方案的死穴在于:试图用一个万能模块(LLM 或人)去判断"这个结果对不对"。这种"二元判断"(对/错)在语义世界里永远会陷入 Precision-Recall 的泥潭——实验 E 的三档模型实测已经证明了这一点。

替代方案的核心是:**不判断对错,只判断风险。** 然后把高风险任务直接踢出 AI 流水线用确定性代码处理;低风险任务自动放行;中风险任务才给人看。

---

## 四层实操架构

全部用确定性代码实现,零 LLM 依赖(除了最底层 Agent 执行本身)。

### 第一层:任务类型强制分流

在任务进入 Agent 引擎之前,由路由层根据任务标签做硬分流:

| 类型 | 判定标准 | 处理策略 |
|------|---------|---------|
| **Type A(可验证型)** | 产出可被正则/编译器/Schema 验证(代码/JSON/SQL) | **全自动。** 编译通过/Schema 校验通过直接放行。不调用任何 LLM 质检员 |
| **Type B(高风险型)** | 涉及金钱、法律、用户隐私、对外发布 | **不做 Agent。** 弹窗提示"该任务需人工处理",AI 仅提供参考草稿,绝不自动执行 |
| **Type C(低风险内容型)** | 内部简报、初稿草拟、头脑风暴 | **全自动放行,** 标记为"草稿"状态,默认置信度 80%,不进入质检队列 |
| **Type D(中风险内容型)** | 给客户看的邮件、对外文档 | **进入 Diff Review,** 不判断内容好坏,只展示修改点 |

**为什么这比"质检 LLM"强:** 因为它在源头就承认了 LLM 的极限。只用 LLM 做它能做的(生成),绝不用 LLM 做它做不好的(判断语义质量)。

### 第二层:差异审核——替代"判断对错"

这是最关键的操作性替代。对于 Type D 任务,不给审核员看"最终结果",而是看"Agent 修改了人类上一版的哪些地方"。

**做法:** Agent 生成内容后,系统强制与"原始版本"或"模板"做字符级/语义级 Diff(difflib 即可,不需要 LLM)。

**审核员界面:** 只高亮显示被修改的段落。审核员只需问一个问题:**"这个改动有没有引入错误?"**

**对比实测数据:**

| 方式 | 阅读量 | 认知负荷 | 时间 |
|------|-------|---------|------|
| 全文判断好坏(500 字新文档) | 500 字 | 高 | 约 60 秒 |
| 差异审核(50 字 diff) | 50 字 | **低** | **约 10 秒** |

核心转变:把开放式判断("这篇文章好吗?")变成了封闭式判断("这段话改错了吗?")。后者的认知负荷低一个数量级。

### 第三层:统计过程控制——替代语义聚类

之前"聚类"失败是因为按语义聚——简报和初稿语义不同,聚不到一起。替代方案:**不按语义聚,按行为特征聚。**

具体做法:记录每个任务的元数据——执行轮数、工具调用次数、输出长度、特殊字符占比。

**异常检测:** 不判断内容好坏,只看行为模式。如果一个任务执行了 15 轮(均值是 5 轮),或输出长度为 0,或全是标点符号(实验里的"句号"和"鸭子"),**直接用代码标记为"疑似异常"**,推送给人。

**优点:** 代码判断,成本为 0。且能抓到"鸭子"(长度异常)和"句号"(特殊字符占比),不依赖语义。

**实测:** 拿实验 E 的 8 个场景跑 SPC 检测(内容长度、特殊字符%等行为特征,统计阈值=meean ± 1.5sd):

| 场景 | SPC 结果 | 实际 | 正确? |
|------|---------|------|-------|
| G1(鸭子·垃圾) | **异常**(高中文占比) | 垃圾 | ✅ |
| G2(句号·垃圾) | **异常**(特殊符 100%) | 垃圾 | ✅ |
| G3(TODO·垃圾) | **异常**(字母 100%) | 垃圾 | ✅ |
| **G4(零用例·垃圾)** | **正常**(同 L4 特征一致) | **垃圾** | **❌ 漏报** |
| L1-L4(合法) | 正常(含 1 个 L2 弱误报) | 合法 | ✅ |

SPC 抓住了格式异常(句号/TODO/鸭子),**但 G4(零用例)和 L4(合法测试日志)在行为特征上几乎一样——SPC 100% 漏报。** 这直接验证了盲区声明:SPC 能抓格式异常,不能抓语义陷阱。G4 类只能靠抽样发现,无法预防。

### 第四层:固定抽样率——替代置信度反馈

之前"置信度 > 95% 放行,抽检 1%"的代价是一套反馈闭环——和缺陷二(闭环校准)里被拆的那套一样复杂。

替代方案:**抛弃置信度。用固定抽样率。**

| 类型 | 处理 | 抽样率 |
|------|------|--------|
| Type A | 全自动,无抽检 | 0% |
| Type B | 强制人工,不做 Agent | — |
| Type C | 全自动放行 | 0% |
| Type D | Diff Review(全部) | 100% |
| 从零生成(无前版/无模板) | 抽样审核 | **固定 5%** |

**为什么不用置信度:** 因为置信度需要一个反馈闭环来校准。而反馈闭环需要 ground truth,需要延迟,需要数据库——这和缺陷二里被批的"元 LLM"是同一类架构复杂度。固定 5% 是拍脑袋的(我承认这一点),但它的数学性质是已知的、可量化的。

## 严苛审视:修正版

方案写到这里,我用前面六刀的那把尺子量了它一遍。

### 审视一:分类不是免费的

Type 划分由业务方完成。但:

- **业务方不知道自己的类型。** 让她给每个任务打标签,她大概率把 70% 都标成 D(怕出事)。D 越多,Diff Review 队列越长,D 的压缩率就回到聚类方案的老路上。
- **任务类型是上下文相关的。** "写一封给客户的邮件"——如果是确认收到附件,可能是 C;如果包含价格承诺和赔偿条款,应该是 B。
- **混合类型:** 一张工单同时包含"内部技术方案(C)"和"附带的对外承诺(B)"。Router 怎么分?

**修正:** 不依赖业务方手动打标。在 MVP 阶段,用两条硬规则自动划分:①如果任务文本包含敏感关键词(金额/合同/赔偿)→ 强制 B;②如果工具调用链中出现"发送/发布/提交"类工具 → 强制人工确认。剩下的默认 C。上线后根据误命中率微调。

### 审视二:Diff Review 的覆盖范围比"编辑型任务"窄

Diff Review 最漂亮的点——把开放式判断变成封闭式差分判断——只在**修改已有内容**时成立。但 Agent 的大量任务是**从零生成**:写一份新的竞品分析简报、起草第一条客服回复。

这两种场景:没有"原始版本",没有"模板",diff 什么?

- 如果 diff 模板,模板到成品的跨度远大于"改了十几个字"——diff 高亮 90% 的内容,等于没高亮。
- 如果退化成全文阅读——就回到了"60 秒判断好坏"的老路。

**修正:** 本方案中的"编辑型任务"精确定义为"已有同一文档的前一版本",不包含"读多份参考文档写新文档"。后者统一归入"从零生成"→ 5% 固定抽样。这是一个诚实的收缩。

### 审视三:5% 抽样率的数学

假设从零生成任务每天 100 件,5% = 5 件被抽到人审。如果某天的真实缺陷率是 20%(包括类 G4 陷阱):

- 抽到至少 1 件有缺陷的概率 = 1 − (0.8)⁵ = **67%**
- **33% 的概率,你当天零检出。**

七天中就有一天有 33% 概率漏掉整天的系统性退化。监控仪表盘依赖连续窗口的统计显著性——如果 7 天里 6 天正常,1 天有 20% 缺陷但抽样漏了,仪表盘不出声。

**修正:** 5% 对于非关键内容可能够用。对于关键内容,上调到 10%–20%,或改为"每 N 件必抽 1 件"的确定性抽样(非随机)。建议上线第一周用 20%,采集历史缺陷率数据后再调整。

### 审视四:敏感操作拦截不是免费的

敏感操作白名单拦截发送邮件/改权限。假设用户的任务是:"写一份客户报价单,如果金额超过 5000,同时发送邮件通知"。Agent 的调用链:写报价单(允许)→ 检查金额(允许)→ 发送邮件(敏感,拦截)。

**拦截发生在 Agent 跑了 3-4 步之后。** 被拦截的步数消耗了推理成本,不是免费的。

**修正:** 敏感工具白名单应当配合"预热检测"——在任务进入 Agent 前,扫描用户请求文本中包含的敏感动词(发/改/删/提),提前做二次确认。不等到执行期扣扳机。

### 审视五:工程投入估算——我犯过同样的错

我原来说 2 人·月可以完成 MVP。和缺陷五(成本账)里犯的错一样——只算了核心模块,没算集成。

实际分解(诚实版本):

| 模块 | 工作量 |
|------|--------|
| Diff UI(可视化+高亮+判断按钮) | 1 人·月(前端为主) |
| SPC 统计器(元数据+阈值+聚合) | 0.5 人·月(后端) |
| 敏感工具白名单+执行期拦截器 | 0.5 人·月(端到端,需侵入 Agent 框架) |
| 监控仪表盘+报警 | 1 人·月(前后端) |
| 固定抽样队列+分配+过期 | 0.5 人·月(后端) |
| **合计** | **3.5 人·月(MVP)** |

比那套 5 人·月的人工 Harness 便宜 **30%**(不是 60%)。这个数字不性感,但真实。

## 诚实收束:这套方案还剩什么

### 解决了什么

- **ROI 倒挂:** Type A 全自动 + C 直接放行 + D 仅审 diff,需要人看的中高风险项占比大幅压缩。3.5 人·月投入,适用场景的月度人工成本节省超过 60%。
- **聚类失效:** 用 SPC 替代 embedding,统计行为模式,代码可验证,成本 0。
- **人犯错:** Diff Review 降低认知负荷,但不能消除。G4 类语义陷阱仍需领域知识。

### 没解决什么

- **抓不住 G4 类语义陷阱。** 靠抽样发现,无法预防。这和靶子的"确定性约束"是最本质的不同——他们声称预防,我们承认检测。
- **依赖人做最终决策。** 在敏感操作和编辑任务中,人不可替代。
- **从零生成任务靠抽样。** 不保证 100% 质量。但用 20% 抽样可以把单日漏检率降到可接受水平。
- **分类准确性问题。** 不依赖业务方打标,但关键词过滤+工具链检测的准确率仍有限,需上线后校准。

### 门槛是什么

- 需要实现:工具调用白名单、SPC 统计阈值、Diff UI、固定抽样队列、监控仪表盘——都是常规 CRUD+正则+统计,没有 LLM 依赖。
- 工程投入:3.5 人·月可完成 MVP。
- 业务前提:接受"高风险必须人工"、"从零生成靠抽样"、"语义陷阱靠发现不是靠预防"——这三个约束是商业决策,不是工程能解决的。
