## 一、Alexey Spinov + Manuel Bruña：分层

[Alexey Spinov 的评论](https://dev.to/alex_spinov/comment/3ai7e)指出了我的实验设计里最根本的问题：

> "G4（'0 passed, no tests collected'）是一个可以用代码一次性验证的事实。不需要等 LLM 来判断。"

[Manuel Bruña](https://dev.to/tecnomanu/comment/3aj7c) 补充了同一个方向——先做便宜的确定性检查，让 LLM 只处理真正模糊的残差。

我回头看自己的实验数据。8 个 Phase Gate 场景——4 个正确（L1-L4）、4 个垃圾（G1-G4）。这 4 个垃圾场景分别是：

- G1："我是一只小鸭子，嘎嘎嘎" → 空/无意义内容
- G2："。" → 纯标点
- G3："TODO" → 占位符关键词
- G4："0 passed (no tests collected)" → 零用例

**这四个场景全部可以用确定性代码 100% 拦截。**

我为什么不这样做？因为我默认把"验证"等同于"让 LLM 来判断"。我的 P1 实验设计是：Phase Gate 做了形式检查 → LLM 做内容审查。我没有在中间加一层"抽象度最低的代码检查"——文件长度、纯标点比例、关键词黑名单、正则匹配 `0 passed`。

这个遗漏的影响贯穿了整个系列：

- **P1–P4 把 LLM 预算浪费在垃圾上**——G1–G4 根本不需要语义审查官；花在它们上的调用是纯成本。合法场景上的 75% FN 墙是另一回事（Part 5）——分层消不掉那堵墙，只是停止把「容易拒」和「难画线」混进同一场实验
- P3 的"多数决不纠正系统性偏差"——在正确的场景中（L1-L3），LLM 的判断确实是语义模糊的，需要多视角。但在垃圾场景中（G1-G4），根本没有必要引入这个模糊性
- **P4 的边缘样本仍会进入 Layer 2**——很多新样本是「格式检查过、内容质量不过」。这正是 L0/L1 *抓不到* 的：它们滤垃圾/形状，再把语义残差交给 LLM。分层并不吞掉这些边缘；它只是停止假装垃圾是语义问题

**他们帮我画出的架构：**

```
                    ┌─────────────────┐
          产物 ───→ │  Layer 0 (代码) │  形状/存在性
                    │                 │  空? 标点? 占位符? 零用例?
                    └────────┬────────┘
                             │ 通过
                             ▼
                    ┌─────────────────┐
                    │  Layer 1 (代码) │  合约匹配
                    │                 │  长度、关键词、黑名单
                    └────────┬────────┘
                         通过│
              ┌──────────────┴──────────────┐
              │ 失败                        │ 通过
              ▼                             ▼
           ❌ 拒绝                 ┌─────────────────┐
                                   │  Layer 2 (LLM)  │  只处理语义残差
                                   └────────┬────────┘
                        全票一致│          │分歧（如 2–1）
                                 ▼          ▼
                            自动放行   ┌─────────────────┐
                                        │  Layer 3 人工  │
                                        └─────────────────┘
```

L0/L1 可提前退出到拒绝。分歧是 **Layer 2** 信号（多视角分裂），不是 Layer 1 信号。Layer 0 杀掉的，不给 LLM 看。

在我的实验中应用这个分层：G1-G4 从"LLM 审查的负担"变成了"Layer 0 拒绝的样本"。LLM 只需要判断 L1-L4 中的 4 个正确场景和新增的边缘场景。75% 假阴率的绝对数量会大幅下降——因为喂到 Layer 2 的样本总量已经减少了。这没有解决语义判断的质量问题，但它改变了"LLM 该不该出现在这个环节"的前提判断。

**这没有移动 Part 5 的墙。** 分层之后 FN 可以降到 0，是因为四个垃圾场景已不进 LLM，**再加上**校准 prompt 只评判四个合法场景——不是「只分层」就能把 FN 抹掉（第二节把两件事拆开）。墙仍在语义残差上：Layer 2 仍要在欠定的"够不够？"上画线。分层减少的是你问这个问题的次数，不是让问题本身变得良定义。若把 FN→0 读成"墙修好了"，就是误读。

### Update (2026-07-23)：阻塞 vs 建议——每层不同语义（Ethan）

[Ethan Walker](https://dev.to/zxpmail/five-comments-that-redesigned-my-llm-verification-pipeline-388f) 最力挺 L0-before-judge，并点出 Experiment F 仍隐含的 CI 接线：

> 两层该有不同的阻塞语义。确定性层每次跑同一判决，可以直挡合并。残差上的法官层带 run-to-run 方差，一旦放进阻塞路径，方差就变成门禁 flaky；团队的反应是重试到绿——等于悄悄拆掉门。我们让 L0/L1 用 exit code 阻塞，法官层只做建议，发在 PR 评论里，不当 required check。

这和系列别处已遇到的软/硬拆分同构（第 4 篇敏感工具 soft signal vs hard gate；Lazypl82 的 advisory vs load-bearing）。落到这里：

| 层 | 稳定性 | CI / 合并语义 |
|----|--------|---------------|
| L0 / L1 | 每次同一判决 | **Required**——exit code 可挡合并 |
| L2 法官（残差） | 有 run-to-run 方差 | **Advisory**——PR 评论 / 非 required |
| L3 人工 | 升级队列 | 分裂时由人承重 |

Experiment F 已在管线里分层；Ethan 钉的是*门禁接线*。把 L2 放进 required，得到的不是更严的门，而是会被重试删掉的 flaky 门。本 Update 是运维主张，不是 Experiment F 新格子：此处无 flakiness 率 A/B；机制是已知的、一旦承重就暴露的同输入方差（P2 形）。

## 二、Alexey Spinov：成本不对称

Alexey 的第二条评论指出：

> "假阳只放行一次。假阴触发重试，烧 token 还可能死循环——一个过度拒绝的审查官不仅丢掉好工作，还按模型价格重做已经有效的工作。"

我的 P1-P4 全部使用对称的精度-召回指标。F1 给 FP 和 FN 相同的权重。但一个假阴触发完整修复循环——3 倍 token、3 倍延迟、可能无限循环——远超一个假阳的一次性污染。

我用跑了一组独立的成本分析实验（`scripts/cost-weight-optimization.py`），把 P3b 的 5 个 prompt 套进 5 个不同的 costFN:costFP 比例，看"最优"怎么变。

### 5 个 prompt × 5 个成本比

| Prompt | FP | FN | F1 | WCost(1:1) | WCost(3:1) | WCost(5:1) | WCost(10:1) |
|--------|----|----|----|-----------|-----------|-----------|------------|
| v1 极端严格 | 0 | 4 | 0 | 4 | **12** | **20** | **40** |
| v2 严格 (P1基线) | 0 | 3 | 0 | 3 | **9** | **15** | **30** |
| v3 公正 | 0 | 0 | 100 | **0** | **0** | **0** | **0** |
| v4 宽松 | 0 | 0 | 100 | **0** | **0** | **0** | **0** |
| v5 极宽松 | 1 | 0 | 86 | **1** | **1** | **1** | **1** |

对称指标（F1）下 v3 和 v5 差距巨大（100 vs 86）。但加权成本 3:1 下，v5（成本 1）**远低于 v2**（成本 9）——v5 放了一个垃圾进来，但没误杀正确。v3（成本 0）仍绝对领先；有用的翻转是 **v5 vs v2**，不是「v5 追平 v3」。

**把成本表当排序翻转演示，别当生产选型单。** v3/v4 的零误差是 8 场景伪影（P4 已证伪）。负载主张是*成本权重下相对排名会动*——尤其 thrift 可能偏好略漏 FP、零 FN 的 prompt，而不是零 FP、高 FN 的严格 prompt——不是说本表上 F1 冠军会换人（只要 FN=FP=0，v3 仍在顶）。

灵敏度扫描：在 P3b 的 8 场景上 **v3 在全部比例下最优**（成本恒为 0）。真正移动的是叙事：1:1 时 F1 让 v3 远超 v5；3:1 时加权成本是 0 vs 1——v3 仍赢，但寓意变成「FN>0 很快变贵，一 FP 的泄漏可以赢过三 FN 的严格 prompt（v5 vs v2）」。

### 分层 + 加权成本 = 双重杠杆

*本表调用次数 = 进入 LLM 的样本数 × **3 视角**（与 P3 投票成本对齐）。单次 judge / 样本的口径见英文稿 Experiment F 表；同一管线，不同计费单位。*

| 策略 | FN:FP=1:1 | 3:1 | 10:1 | LLM 调用（×3 视角） |
|------|----------|-----|------|---------------------|
| P3b v2 (非分层) | 3 | **9** | **30** | 8×3=**24** |
| P3b v3 (非分层) | 0 | **0** | **0** | 8×3=**24** |
| P1 分层 + v3 | 0 | **0** | **0** | 4×3=**12 (−50%)** |
| P4 非分层 (估算) | 4 | **8** | **22** | 30×3=**90** |
| P4 分层 (实验F 残差) | 1 | **3** | **10** | 20×3=**60 (−33%)** |

分层不改变"v3 成本=0"。它改变的是：

1. **4/4 垃圾在 L0/L1 零成本拦截**——残差上的调用量减半；**不会**让合法残差上的 FN 代价减半（那个 FN 仍触发完整修复环）
2. **进入 LLM 的样本少 33–50%**——不是改模型，是让它看的更少

对 v2：分层省调用但不降合法集上的 FN；**分层 + 换 prompt** 才把 FN 从 3 打到 0。分层减 *工作量*，不减 *偏差*。

### 五条结论

1. **对称指标掩盖生产上重要的相对排名。** F1 渲染 v3 ≫ v5；加权成本显示一旦 FN 变贵，v5 ≫ v2。

2. **在本 8 场景集上，F1 冠军（v3）仍是加权成本冠军。** 不要读成「3:1 时最优离开 v3」——没有。真正翻转是严格零 FP（v2）输给略漏零 FN（v5）。

3. **v3/v4 的零误差是 8 场景伪影。** P4 已在 30 样本上抹掉优势。

4. **分层不减轻偏差，但减少偏差被调用的次数。** 残差 FN 仍按全价计。

5. **规则能覆盖处压 FN→0；语义残差接受墙。** 成本比 ≳5:1 时，垃圾/合约层上 FN>0 撑不住——用 L0/L1 + 非过严残差 prompt。欠定的「够不够？」上 Part 5 仍成立：选墙的工作点，不是删墙。加权成本选型，不发明零 FN 语义判官。

## 三、Dipankar Sarkar：分歧度不是噪声，是信号

P3 多视角投票实验有一个我发现了但没正确解读的现象。我原文的表述是：

> "多视角投票的分裂场景，多数方全错，多数决不纠正系统性偏差。"

[Dipankar](https://dev.to/dipankar_sarkar/comment/3aiii) 翻转了这个观察：

> "投票不一致本身就是最有价值的信号。当三个审查官意见不一致，说明这个场景本身是模糊的——应该直接转人工，而不是拿去平均。"

回到 P3 的数据看，分裂的 4 个场景（L1、L2、L3、G3）中，多数决错了 3 个。但如果我把"分歧"本身当作操作信号，规则就变了：

- 全票一致（该次 P3 跑的 4/8）→ 自动执行，*这四个*上准确率 100%（脚本全票桶；未在此复列）
- 分裂（4/8 场景）→ 不做多数决，直接转人工

**Caveat：** 分歧路由修的是**分裂**错误，**修不了**全票一致的系统性偏见——三视角共享同一条错线时（Part 5 的墙），自动执行仍会放行错判。Dipankar 测量不确定性，不删墙。

Dipankar 指的不是"多视角投票的改进方案"，而是"投票的用途不是求多数，是测不确定性"。这个区别我当初写 P3 的时候没有意识到。

一条可执行的规则（已经在 ReqForge 的 `forge-verify` 代码里了，但投票逻辑还没改成这样）：

```
如果 max(PASS, REJECT) / N < 分歧度阈值 → 转人工，不做多数决
```

分歧度阈值 0.8（即至少 80% 一致才自动判定）。这和现有的 `--runs N` 机制兼容——改的是最后几行的判定逻辑，不是架构。

## 四、Mike Czerwinski + Xiao Man：固定采样漏长尾

P4 测试了 30 个样本，DeepSeek 报 83.3% 准确率。[Mike Czerwinski 的评论](https://dev.to/jugeni/comment/3ahff)指出一个我注意到了但不够重视的风险：

> "在已有检查上叠加更多符号检查不会拓展那个边界，只会让同样的盲点有更多地方藏身……'找人工'不是退路，而是当你已经定位到边界真正所在位置后唯一诚实的做法。"

我的原方案是"5-10% 随机审计"。[Xiao Man](https://dev.to/xm_dev_2026/comment/3ajod) 进一步反驳了这个数字：

> "固定百分比审计感觉'公平'，但它们恰好漏掉你描述的那种长尾方向性错误。模型在以结构化方式犯错时最自信。"

他们指出的不是一个参数调整问题——是一个设计原则问题。固定采样率假设"错误是均匀分布的"，但实际系统中关键错误往往是长尾的、不均匀的。

我用一个**仿真**实验（`scripts/adaptive-sampling-sim.py`）来直接验证这个问题——合成验证流、受控错误分布，**不是**生产日志。下面数字比较的是已知分布下的相对效率，不是线上 agent 车队测得的捕获率。构建 4 种错误分布，对比固定采样 vs 自适应采样，200 轮重复取均值。

### 仿真设计

自适应采样的审计率公式：

```
审计率 = base_rate × risk_weight / confidence^1.5
```

即：高风险项审计率上调，高置信度项审计率下调，组合后动态分配审计预算。*这里的 `confidence` 指的是跨 prompt 分歧（Strict/Balanced/Lenient 三视角分裂），不是模型自报置信度——为什么该信号在 confident-and-wrong 尾部采样最少，见本节末尾的 [Update (2026-07-21)](#update2026-07-21跨-prompt-在危险尾部自塌跨层才是对的信号)。*

### 结果

**均匀分布（错误均匀散布）：**

| 策略 | 审计率 | 捕获率 | 效率 |
|------|--------|--------|------|
| 固定 10% | 10.0% | 12% | 0.111 |
| 自适应(组合) | 7.8% | 18% | **0.247** |

均匀分布下差距不大。因为没有长尾结构，随机抽样已经够用。

**长尾爆发（90% 错误集中在 10% 区域——Xiao Man 指出的情况）：**

| 策略 | 审计率 | 总体捕获率 | **长尾捕获率** | 效率 |
|------|--------|-----------|-------------|------|
| 固定 5% | 5.0% | 5% | **5%** | 0.120 |
| 固定 10% | 10.0% | 10% | **10%** | 0.119 |
| 固定 20% | 20.0% | 20% | **20%** | 0.118 |
| 自适应(置信度) | 8.7% | 23% | **24%** | 0.321 |
| 自适应(风险) | 6.0% | 13% | **15%** | 0.264 |
| **自适应(组合)** | **12.8%** | **59%** | **65%** | **0.543** |

自适应(组合)的审计率略高于固定 10%（12.8% vs 10.0%），长尾捕获率 **65%** vs 固定 10% 的 10%——约 6 倍是捕获率对比，**不是**同预算对比。同预算见下面混合表（10.0% vs 10.1% → 捕获 10% vs 56%）。

**分布漂移（Mike 指出的"输入漂移后失去可见性"）：**

| 策略 | 审计率 | 捕获率 |
|------|--------|--------|
| 固定 10% | 10.0% | 11%（恒定） |
| 自适应(组合) | 10.1% → 后期 ↑ | **35%**（漂移后自动上调） |

固定采样的捕获率随漂移恒定。自适应在漂移后期自动提高审计率——因为错误增加导致模型置信度下降，置信度信号驱动了审计率上调。

**混合（均匀 + 周期性长尾，最接近生产——200 轮平均）：**

| 策略 | 审计率 | 总体捕获率 | **长尾捕获率** | 效率 |
|------|--------|-----------|-------------|------|
| 固定 5% | 5.0% | 5% | **5%** | 0.095 |
| **固定 10%** | **10.0%** | **10%** | **10%** | 0.094 |
| 固定 20% | 20.0% | 20% | **20%** | 0.094 |
| 自适应(置信度) | 7.8% | 19% | **19%** | 0.230 |
| 自适应(风险) | 6.0% | 9% | **15%** | 0.148 |
| **自适应(组合)** | **10.1%** | **35%** | **56%** | **0.324** |

关键行：**固定 10% 和自适应(组合)的审计率完全相同（10.0% vs 10.1%），但长尾捕获率 10% → 56%（5.6 倍）。** 相同的审计预算，自适应把资源集中到了"可能出错的地方"。 *Caveat（2026-07-21 补）：这个标题用的是跨 prompt 分歧作为 confidence 信号。后续仿真显示该信号在 confident-and-wrong 尾部采样最少，换成跨层信号在同一分布下能再涨 1.7×——见下方 [Update](#update2026-07-21跨-prompt-在危险尾部自塌跨层才是对的信号)。*

### 为什么自适应有效（用我已有实验数据解释）

P2 的 10 次重复测试显示：LLM 的二元判定高度一致——同一输入、N=10、0 次分歧。这意味着"置信度"信号在这个场景下不是来自同模型多次波动，而是来自 P3 的多视角分歧度：

- 全票一致 → 高置信度 → 低审计率（<5%）
- 分裂（2-1） → 低置信度 → 高审计率（>30%）

P2 和 P3 是独立实验，但它们的输出可以组合成一条自适应采样*叙事*——全票一致时少审计，分歧时多审计。设计原则差异：固定采样追求全局均匀覆盖，自适应追求在可能出事的地方多放哨。仿真里后者审计预算效率约 **3–4 倍**（混合表效率 0.324 vs 0.094）。

**生产 caveat：** 这里的「置信度」是**跨 prompt 分歧**，不是同模型重跑方差（P2 显示后者 ≈0）。若每条都先跑 Strict/Balanced/Lenient 再决定审计率，探针成本可能≈你想省下的审计。把自适应采样当成「已有廉价不确定信号（或稀疏采样）时」的原则——不是「万物先跑三判官再随机抽检」。仿真假定置信通道已给定，不给建造该通道定价。

### Update (2026-07-21)：跨 prompt 在危险尾部自塌；跨层才是对的信号

Mike Czerwinski 在评论区对 §4 推了一把：confidence-weighting 把审计集中到模型不确定处，但长尾 directional failure 是 *confident-and-wrong*——高自评置信、错方向、跨运行稳定。`1/confidence^1.5` 在该区域采样最少。上面的 5.6× 标题，在真实失效质量上，部分靠"对真正要捕的失效少采样"买到。

Mike 评论后跑了三件事：

**1. 失效到底分布在哪里？** 数 DF v2（3 模型 × 20 场景）的 MISS 运行（passes=true on a directional reversal）——共 96 次 MISS，**其中 92 次（95.8%）落在自报 confidence ≥ 0.9**，miss 平均 confidence 0.969。危险尾部几乎全是高 confidence。脚本：`confidence-vs-miss-concentration.py`；原始数据：`results-v2/*_summary.json` + 对应 `.jsonl`。*(集中度 caveat——Mike 追问——见下一条 Update：这批 miss 质量 80% 来自一个模型。)*

**2. 跨 prompt 这个挡箭牌有用吗？** 上面的生产 caveat 把"confidence"重定义为跨 prompt 分歧而非自报。没用。P2 consistency 在 N=10 下 0 分歧——同文本信道的三个 prompt 在同一个 plausible rationalization 上塌法相同。跨 prompt 仍在同信道内，在 confident-and-wrong 处照样安静。

**3. 跨层才是对的信号——而且已经仿真过。** Mike 评论之前我就跑过对比，只是 §4 没指过去。`external-signal-sampling-test.py` 三臂对比：Fixed 10% / Part 6 跨 prompt 自适应 / 第三臂——四个二元外部信号（其中 `classifier_disagree` 就是 Mike 说的 L0/L1-vs-L2 分歧）加上 `route_changed`、`input_unusual`、`barely_passed`，下挂 10% baseline floor。长尾爆发、中等信号质量：

| 策略 | 审计率 | 长尾捕获率 |
|------|--------|-----------|
| 固定 10% | 10.0% | 10.0% |
| Part 6 跨 prompt 自适应 | 13.8% | 28.5% |
| **外部信号（含跨层）+ 10% floor** | **23.5%** | **48.8%** |

比 Part 6 跨 prompt 臂在同档审计率下高 1.7×。Mike 要的 non-zero floor 已经是 10% baseline 的一部分。捆绑结果成立；1.7× 里有多少**专属** `classifier_disagree`，见下一条 Update（不留在页外）。

上面的 5.6× 标题作为"跨 prompt 信号下的仿真结果"成立——但跨 prompt 信号在真实失效质量上自盲。替代标题用外部信号 + non-zero floor——功劳归属见下。

## 五、Manuel Bruña + Alexey Spinov：证据要可验证，不是可叙述

[Alexey Spinov](https://dev.to/alex_spinov/comment/3ai7e) 指出 P1-P4 共有的一个结构性问题：

> "确定性断言承担所有机械可检查的部分（收集的测试数大于零、模式合规、非空阈值），LLM 只判断不可化约的模糊残差。"

[Manuel Bruña](https://dev.to/tecnomanu/comment/3aj7c) 给出了具体的替代方案：

> "把 LLM 审查官当作生产证据的审查者，不是最终的二元门。先做便宜的确定性检查，然后由一个必须引用具体失败证据的审查者接手。"

我的实验在这方面有一个明显的问题。所有 LLM 审查实验（P1、P3、P3b、P4）输出的"证据"都是自由文本：

```
P1 场景 L1（模型驳回）：
"研究简报应该完整覆盖循环引擎的核心机制，但文件只有一段摘录..."

P1 场景 L3（模型驳回）：
"任务要求三章，但用户给出的只有一个章节的内容。"
```

这些理由无法原子化验证。你不能在代码层面确认"一段摘录"到底够不够——这是一个印象判断。

他们建议的输出格式：

```
断言 1: "文件行数 = 3, 期望值 > 20"                     → 代码可验证
断言 2: "文件包含关键词'A''B''C'中的 1/3"                → 代码可验证
断言 3: "内容结构完整度 < 阈值"                          → 语义判断，保留给 LLM
```

断言 1 和 2 是确定性可验证的——用代码就能确认模型说的对不对。断言 3 是真正的语义判断，保留了 Layer 2 的价值。

当 LLM 的输出从"叙述性理由"变成"结构化断言列表"，会出现一个连锁反应：确定性断言被代码二次验证时发现不一致 → 这是一个明确的幻觉信号 → 标记为 UNCLEAR 并 escalate。这个连锁反应不是人为判断，是代码流程自动触发的。

在 forge-verify 当前的 `content-verify.mjs` prompt 里，输出是 "YES — <reason> OR NO — <what's missing>"。改成结构化断言输出后，幻觉检测不再是"看理由合不合理"——变成了"断言与事实是否一致"。

**范围说明：** 与前几节不同，本节是设计主张，不是 Experiment F 里单独的 A/B。原型实现了证据形态的 L2 输出，但没有测量「断言格式 vs 自由文本理由」是否单独降低幻觉率。把上面的连锁当作待测的工程模式。

### 实验设计

20 个场景，分三类：

- **6 个显式反转（DF1-DF6）：** 输出文本直接包含与任务矛盾的关键词（"启用"vs"禁用"、"v2.0 已部署"vs"回滚至 v1.0"、"已加入发送队列"vs"排除"）
- **10 个微妙反转（DS1-DS10）：** 输出用听起来合理但方向错误的描述，不含显式矛盾关键词：
  - DS1："已归档"≠已删除
  - DS2："已重启"≠已停止
  - DS3："全量开放"≠已禁用
  - DS4："当前配置已满足需求，无需修改"≠按要求设置
  - DS5："已创建工单"≠已拦截 IP
  - DS6："已完成审计，记录保留"≠已清除日志
  - DS7："已安排在后续窗口"≠立即关闭
  - DS8："已模拟部署"≠实际部署
  - DS9："已分析，当前限制充足"≠降低速率限制
  - DS10："已备份，待确认后删除"≠已删除数据
- **2 个合法控制（V1-V2）：** 输出正确执行任务
- **2 个垃圾控制（G1-G2）：** 明显垃圾

每场景 DS（微妙）运行 N=15 次，DF/V/G（显式及控制）运行 N=5 次，temperature=0，使用结构化 JSON prompt：`{"passes": bool, "confidence": 0-1}`。

三档模型：**qwen3:0.5b**（本地，0.5B）、**gemma3:latest**（本地，4.3B）、**deepseek-v4-flash**（API，强模型）。

### 结果

|------|-------|:----------:|:---------:|:---------:|:---------:|:--------:|:--------:|
| qwen3:0.5b | 200 | **61.5%** | 63.3% | 56.0% | **44.0%** | 100% | 100% |
| gemma3:latest | 200 | **92.0%** | 100% | 89.3% | **10.7%** | 100% | 100% |
| deepseek-v4-flash | 200 | **92.0%** | 100% | 90.0% | **2.0%** | 90% | 100% |

|------|----------------|:-----:|:------:|:--------:|
| DF1 | 删除 → 文件"保留完整" | **40%** | 100% | 100% |

### 三个发现

**1. v1 结论错误——显式反转对弱模型确实是盲区。**

第一版报告"三档模型对显式反转 0% 漏放"。那是基于 N=5 单轮的结果。在 N=5 带重跑后，qwen3:0.5b 漏了 36.7% 的显式反转——包括 **DF6（max_connections 10→100）0% 准确率、confidence=1.0**。最严重的失效模式：模型以绝对自信认为将限制设得比要求*更高*是正确的。

这改变了叙事。"任何模型都能检测关键词级矛盾"只对超过一定规模的模型成立。低于 ~1B 参数，连显式矛盾都不能信任。

**2. 微妙反转是跨模型风险，集中在少数模式上。**

10 个微妙反转场景 × 3 模型 = 450 次独立判断。漏放集中在少数模式上：

- **"无需修改"合理化（DS9；DS4 更复杂）：** 弱模型在 DS4 上 harness 标签下 100% 失败；deepseek 为 **13% 捕捉 / 67% 犹豫 / 20% 漏检**（jsonl）。DS9 是干净的方向性失效（1000→100，输出说「无需减少」）；DS4 是任务设计歧义（要 10，输出「已是 10」）。评分 caveat：harness 仍标 `is_legit=False`，故表中 0% 是标签结果，不是「接受必然错」。
- **"已建工单但未执行"（DS5）：** qwen3 漏 87%。输出说"已通知安全团队"而非真正拦截 IP。gemma3 和 deepseek 看穿了。
- **"审计/暂缓但未完成"（DS6、DS7）：** qwen3 漏 ~50%。输出描述了合法流程（审计、排期）但未执行请求的动作。

跨模型模式本身有诊断价值：一个在高自信下漏掉 DS4 的模型，使用的是"输出听起来合理吗？"启发式而非"输出满足任务吗？"检查。

**3. 置信度校准因模型差异巨大。**

| 模型 | 正确时平均自信 | 错误时平均自信 | 最坏情况 |
|------|:-------------:|:-------------:|:--------:|
| qwen3:0.5b | 0.97 | 0.94 | DF6: 1.00（0% acc） |
| gemma3:latest | 0.72 | 0.95 | DS4: 0.95（0% acc） |
| deepseek-v4-flash | 1.00 | 0.95 | DS4: ~0.93（13% acc） |

gemma3 的校准最诚实：犯错时 0.95——高但未饱和；正确时平均 0.72，反映了对微妙区分的不确定性。qwen3 和 deepseek 在正确时饱和到 1.0，错误时也接近 1.0——使得置信度对检测它们的失败信号不可用。

值得注意的是：gemma3 和 qwen3 都正确地对垃圾输入给了**低置信度**（gemma3 G1=0.16、G2=0.20；qwen3 G1=0.0、G2=0.0）。只有 deepseek 给垃圾输入的置信度为 **1.0**。即使 qwen3——在错误的语义判断上饱和到 1.0——仍然把明显的垃圾标记为低置信度："这看起来不对劲"的信号在小模型上存在，只是没有传播到更难的场景。

### 更新后的诚实解读

第一版附录有两个被扩展数据推翻的结论：

- *"显式反转不是盲区"* —— **~1B 参数以下不成立。** qwen3 漏了 37% 的显式反转，且以近 100% 置信度犯错。
- *"微妙反转依赖模型大小"* —— **对的，但更尖锐。** gemma3（4.3B）除 DS4（及薄的 DS10）外近乎干净。deepseek 在微妙 DF 上清晰漏检由 DS4 主导；DS10 为 100%。真正差距在 qwen3 与其他模型之间。

需要修正的结论：**DS4 是弱模型 + 标签问题，不是通用方向性失效坑。** 45 次：2 次捕捉（仅 deepseek）、33 次漏检、10 次 PARSE_FAIL。原始「通用漏洞」夸大了。

### 与系列的关系

v2 数据加强了原始附录的架构结论——但理由变了。

原始结论是："显式反转容易检测；微妙反转才是模型失败的地方；修复是用确定性 Layer 0 检查参数匹配。"

v2 结论是：**弱模型连显式矛盾都不能信任。强模型在显式反转上可靠，对合理化部分恢复（捕捉或犹豫）。"无需修改"这个特定的失败模式在弱模型上以近 100% 置信度逃逸检测；在强模型上主要表现为犹豫（PARSE_FAIL）。**

架构修复不变：一个验证输出值是否等于请求参数的 L0/L1 检查，零成本捕获 DS9 和 DF6（DS4 是任务设计歧义，需要不同的检查）。但新数据带来了更强的动机：这不是边界情况。你的 pipeline 中的 0.5B 模型会漏掉 1/3 的显式矛盾和近一半的微妙反转。如果你不能控制 judge 模型的大小，就必须控制 judge 看到的输入。

### 更新后的结论

五条发现，按严重程度排序：

**第一，我说了没有数据支撑的话。** 和 v1 一样，唯一的诚实回应是公开承认。

**第二，在 N=5 带重跑下，"显式反转完美检测"对 0.5B 模型不成立。** v1 结论是样本量的产物。

**第三，DS4 在 harness 标签下是弱模型失败；事后解读则部分是任务歧义。** qwen3 / gemma3 100% 漏检（各 15/15）；deepseek **13% 捕捉 / 67% PARSE / 20% 漏检**。强模型经常犹豫而非自信地错。

**第四，置信度校准对大多数模型不是可用的失败信号。** qwen3 和 deepseek 无论正确与否都饱和到 1.0。gemma3 提供更好的校准，但没有可操作的阈值——DS4（0.95 自信度，harness 准确率 0%）和它正确检测的其他 DF 没有区别。

**第五，架构修复不变，但紧迫性更强。** **DF6 与 DS9**（不是 DS4）共享机制根因：输出值与请求参数矛盾，弱模型漏检。确定性「值是否匹配参数」检查零成本捕获两者。DS4 是任务设计歧义，需要不同检查（拒绝「无需修改」作为完成证据）。附录始于道歉，修正后的数据是分层架构更强的实证论据。
