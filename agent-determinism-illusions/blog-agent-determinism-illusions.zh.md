# 我用四组数据,拆穿了一篇 7000 行的"生产级 Agent"文章

刷到一篇爆款技术文,标题很硬:《ReAct 只是起点,生产级 AI Agent 必须要有确定性的 Loop Engineering》。

文章的核心主张一句话:**ReAct 循环在生产环境会翻车,要在 LLM 的不确定性之上叠加多层确定性约束。** 然后给出一套"循环引擎"设计——Pre-AL Gate、LLM-as-Judge、Phase Gate、决策状态机,声称基于 7000+ 行 Rust 生产代码。

这个**方向是对的**。Agent 循环确实需要工程化的兜底,而不是让 LLM 自己喊停。文章把"自闭环 Agent"从学术幻想拉向工程现实,这个判断值钱。

但问题是——文章反复用"确定性""客观事实""代码否决 LLM"这些词建立安全感。这些词背后到底站不站得住?

我没有去争论,直接跑了三组实验。结论是:**它用来建立"确定性"的三个核心机制,每一个都只做到形式上的确定,在语义层全部失效。**

下面是数据。

---

## 先说公道话

文章最值钱的是**问题意识**:ReAct 循环没有终止条件、不处理中断、不防空转,这三个缺陷是真实的。

提出的方向——"用确定性约束包围 LLM 的不确定性"——也是对的。

问题不在方向,在落地。文章把三个具体的确定性机制当成了已解决的答案,而它们的实际表现,经不起实测。

我要测的就是这三个:

1. **词汇重叠度阈值**(判断用户插嘴是新任务还是补充)
2. **温度 0 的评估器**(判断 Agent 做完没)
3. **Phase Gate**(判断任务完成是客观事实)

三个实验,全部用文章**自己描述的方法和参数**,证伪文章**自己的断言**。

---

## 幻象一:词汇重叠度 = 语义?

用户在循环第 5 轮插了句话:"算了,改成 XXX。"这是旧任务的补充,还是全新任务?

文章的方案:算"词汇重叠度",配两个固定阈值——**≥0.24 判同任务,≤0.08 判新任务**,中间交 LLM 裁决。并声称"80% 用代码秒判"。

听起来很工程。但词汇重叠只看字,不看义。我构造了 30 对带标注的样本,套它的阈值,用三种分词跑。

**结果:硬误判率 50%。**

最狠的几个:

> 当前任务:"继续写那篇循环引擎的文章"
> 用户插嘴:"**删掉**那篇循环引擎的文章"
> 重叠度 **0.615 → 判为同任务**

用户说"删掉",引擎判定"和写文章是同一件事",于是**继续写**。反向操作被当成延续——这是线上事故级的误判。

> 当前任务:"修复支付页面的 bug"
> 用户插嘴:"结账那页报错了你看看怎么回事"
> 重叠度 **0.000 → 判为新任务**

人类一眼就知道是同一件事,Jaccard 给 0。同义改写全军覆没——6 对全错。

跨语言更彻底。中英对照的 6 对同任务,重叠度全是 0.000,全部被判成新任务。双语工作环境里这套机制**当场报废**。

文章辩护者可能说:"你看,代码敢判的有 90%,超过我们承诺的 80%。"

这是偷换。"80% 秒判"的隐含承诺是"80% 的情况**判对**"。实际是:代码下了判决,但**一半判错了**。正确口径是——代码敢判的 27 例里,只对 12 例,**准确率 44%**。

把"判了"当"判对了",是这套机制最危险的话术。

这套阈值只在"易样本"上有效:同任务高重叠、新任务低重叠,12/12 全对。而同义改写、跨语言、反义这三类"常见但难"的情形,16 个里 0 个对。**强烈暗示阈值就是在易样本上调出来的**,换个稍微复杂的样本集立刻崩。

---

## 幻象二:温度 0 = 确定性?

文章说评估器温度设为 0.0,"输出几乎完全确定",因为"同一个输入,评估结果必须尽可能一致"。

这个断言可以一句话测:同一 prompt、温度 0、跑 20 次,看输出一不一致。

我在 GLM-5.2 上跑了三类 prompt,每类 20 次。

**结果:开放式输出只有 70% 一致,30% 发散。**

| Prompt 类型 | 完全一致率 | 互异版本数 |
|------------|----------|----------|
| 数学(最稳定) | 100% | 1 |
| 结构化列举 | 95% | 2 |
| 开放式创意 | **70%** | **5** |

开放式那条最致命——同一个 prompt、温度 0、20 次里冒出 5 个不同版本,最低相似度 **0.198**:

> "Always head Northbound for your daily cup of exceptional coffee."
> "Premium coffee for the journey ahead."

两句话几乎没共同字符。**而 LLM-as-Judge 评估器输出的正是这类开放文本**——done / phase_done / reason / evidence。

文章说"评估器不是创造性写作,是判断,所以温度必须是 0"。但评估输出的 reason、evidence 字段本质就是开放的,实测发散度跟创意 prompt 一个量级。

连"结构化列举"都不稳:5 个形容词换了顺序。如果 evidence 字段是个列表,顺序一变,下游 JSON 就变,决策就可能变。

唯一 100% 确定的是"17×23=391"。这恰恰说明:**温度 0 的确定性只在答案空间极窄时成立**。一旦输出有任何开放度,确定性就崩。文章把"窄域特例"当成"普遍性质",是过度泛化。

评估器的可复现性是整套循环引擎的根基。评估不稳 → Phase Gate 拿到的 done 信号不稳 → 决策状态机不稳。地基一抖,上面十层确定性约束都站在抖动的基础上。

(只测了 GLM-5.2 一个 provider。但文章断言是普遍化的,单 provider 证伪已足够。OpenAI temp 0 的非确定性是官方文档和大量独立测试的已知结论,补测只会加强不会削弱。)

---

## 幻象三:Phase Gate = 任务完成?

这是文章最自信的一句:**"把任务完成从 LLM 的自我宣称,变成了可验证的客观事实。"**

Phase Gate 检查四类:脚本是否 exit 0、文件是否存在、文件数量够不够、用户确认记录在不在。全部用代码,全部检查"客观事实"。

问题是——**这些检查验证的是"动作发生了",不是"结果是对的"。**

我照文章描述实现了 Phase Gate,构造了 8 个场景:4 个内容正确,4 个内容是垃圾但符合 Gate 检查。

**结果:Phase Gate 通过率 100%,内容正确率 50%,假阳率 50%。**

四个假阳样本,自己说话:

| 任务 | 产物实际内容 | Gate 判定 |
|------|------------|----------|
| 写循环引擎研究简报 | **"我是一只小鸭子,嘎嘎嘎。"** | ✅ 通过 → "完成" |
| 写至少覆盖三机制的初稿 | **"。"**(一个句号) | ✅ 通过 → "完成" |
| 生成 3 个章节文件 | **3 个文件全是"TODO"** | ✅ 通过 → "完成" |
| 运行测试确保通过 | **`0 passed (no tests collected)`**,exit 0 | ✅ 通过 → "完成" |

鸭子、句号、TODO、零用例——Phase Gate 全部放行。它对内容正确性零区分力。

这不是我实现错了。文章描述的 4 种检查**本来就不读内容**,任何忠实实现都有同样的盲区。脚本 exit 0 只表示进程没崩,不代表结果对;文件存在只表示路径在,不代表内容符合需求。

文章把"文件存在/脚本跑过"包装成"任务完成",是论点的过度延伸。真相是:Phase Gate 把**"动作发生了"**变成了客观事实,没有把**"任务完成了"**变成客观事实。两者之间有一道它跨不过去的语义鸿沟。

那道鸿沟叫**内容质量**——恰恰是生产系统里用户最关心的。

---

## 三刀合一

文章的核心纲领句是:"在 LLM 的不确定性之上叠加确定性约束。"

现在三条"确定性"全被实测打穿:

| 支柱 | 文章断言 | 实测 | 状态 |
|------|---------|------|------|
| 词汇重叠 = 语义 | "80% 秒判" | 50% 误判,准确率 44% | ❌ |
| 温度 0 = 确定性 | "几乎完全确定" | 开放输出 70% 一致 | ❌ |
| Phase Gate = 任务完成 | "可验证的客观事实" | 假阳率 50% | ❌ |

地基三层都漏,上面十层约束全部站在漏水的根基上。

文章 7000 行 Rust 大概率是真的。但它们守的是**符号层**的门——字符串匹配、文件路径、exit code。语义层(意图、内容、质量)仍然裸奔。

---

## 为什么这套文章能火

它精准踩中了"做过 demo 但没踩过生产坑"的读者的焦虑。对没在生产里跑过 LLM 系统的人,这套机制显得极有分量——因为他没见过这些 practice,也不知道它们在语义层失效。

而真正做过生产系统的人,读完只会觉得"名字起得比内容漂亮":Pre-AL Gate 就是 prompt 注入状态,LLM-as-Judge 温度 0 是评估器常识,"确定性优先"是 try/catch 加字符串匹配,Phase Gate 是校验逻辑,10 条优先级是一串 if-else。机制都对、都该做,但给每个都冠以专有名词,营造出"独创体系"的印象——这是 rebranding,不是技术创新。

更硬的伤:文章开头说"不是伪代码,不是概念图",全文却零行真实代码,只给函数名、常量名、参数值。这些是标识符,不是代码。承诺没兑现。

反复拿来当"生产级"物证的是"7000+ 行"——出现三次。但行数是质量最差代理指标。真在生产跑的系统,该拿出来的是 SLO 数据、故障复盘、压测曲线,不是行数。

## 第四刀:我也撒了谎(拆我自己)

这篇文章的前三刀拆的是靶子文章的"确定性"三支柱。数据说话,三根全断。

但写到这里我必须说实话:我在那三刀后面准备了一套"建设性方案"——embedding 升级词汇重叠、多票评估修补温度 0、质检员 LLM 补充 Phase Gate。我当初觉得这样写能让文章从"批评"升级到"建设"。

**我错了。这套方案和靶子文章犯了同一个病:用复杂的工程假装有语义解。**

我说服自己跑了一组实验来确认。不是跑靶子那边,是跑我自己这边。我用 Qwen3-embedding:0.6b (一个真正的神经 embedding 模型,1024 维)跑了一个完全同样的"同义 vs 反义"分离测试。

结果:

| 类别 | 均值 | 最小值 | 最大值 |
|------|------|--------|--------|
| **同义**(应高) | **0.766** | 0.490 | 0.977 |
| **反义**(应中低) | **0.739** | 0.582 | 0.881 |
| **无关**(应低) | **0.326** | 0.237 | 0.404 |

同义(0.766)和反义(0.739)**只差 0.026——重叠到几乎无法区分。**

"优化代码性能"和"别优化代码性能"的余弦是 0.881——超过了 12 个同义对中的 10 个。

"做一个用户登录注册的功能"和"加一下账户鉴权那块"(同义)的余弦是 0.490——低于几乎所有反义对。

**神经 embedding 能做的唯一分离是"相关 vs 无关"——同义/反义均值 0.75,无关均值 0.326。但一旦话题相同、方向相反,embedding 和 Jaccard 一样失效。**

至此,从字符到统计向量到神经向量,整条分离链全部实测失败:

| 方法 | 同义余弦 | 反义余弦 | 差距 | 结论 |
|------|---------|---------|------|------|
| Jaccard(实验 B) | 误判率 50% | — | 无法分离 |
| TF-IDF char 2-gram | 0.072 | **0.222** | **方向反** | 不行 |
| Qwen3-embedding | 0.766 | 0.739 | 0.026 | 不行 |

我那套"embedding 升级"方案,在这组数据前不成立。我删掉它,换成诚实的版本。**

## 诚实的结论:当前技术栈下,这个问题没有工程解

靶子文章的三个"确定性"支柱全崩:

- **词汇重叠**不能判断任务连续性(实验 B,50% 误判)
- **温度 0**不能保证评估一致性(实验 A,开放输出 70% 匹配)
- **Phase Gate**不能验证任务完成(实验 C,假阳率 50%)

我试图用 embedding、多票、质检员去补,但实验证明:

- **embedding 也不能分离同义和反义**——话题相同时,同方向和反方向的向量几乎一样
- **另一层 LLM 不解决前一层的不可靠**——质检员本身也会幻觉,只是把问题平移了一层

**所以:当用户在当前话题下说了一句方向不明(新任务还是补充?同向还是反向?)的插话时,工程上不应该用算法单方面判定。检测到话题重叠后,直接问人,不要硬判。**

这不是怂。这是目标函数的诚实选择:"正确性"的优先级高于"自主性"。如果你追求无人值守的自主 Agent,这套东西——无论是靶子的还是我的——当前都做不到。如果你必须保证不误判,人工确认是唯一已知的策略。

**"LLM 执行符号层操作 + 人覆盖语义判断"不性感,但它不撒谎。**

## 落地前的那个问题

如果你看完那篇文章正打算实现一套类似系统,先问自己一句:

**你的任务产出,能否被代码客观验证其"正确性"——而不是"存在性"?**

如果答案是"不能"(大多数内容型、分析型、对话型任务都是不能),那篇文章大部分方案对你无效。你需要的是强人工审核、跨模型交叉验证、用户反馈闭环,而不是文件存在性检查。

如果答案是"能",也请务必自己重调参、重设计验收,并预留大量人工回退通道。**切勿照搬 0.24/0.08,切勿相信温度 0 给你确定性,切勿以为 Phase Gate 过了就是任务完成。**

这三个"切勿",每一个背后都有上面的实测数据撑着。
